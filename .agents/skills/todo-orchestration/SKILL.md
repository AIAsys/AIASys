# Todo 清单编排与执行

规范复杂任务中 Todo 的创建、执行、更新和汇报行为，确保 AI 能够独立、完整、有序地推进任务，不因中间步骤而频繁打扰用户。

---

## 使用场景

以下情况必须启用 Todo 清单：

- 任务涉及 3 个以上独立步骤
- 任务需要跨多个文件修改
- 任务需要调研 + 实现 + 验证的完整闭环
- 任务可能跨越多次 AI 交互才能完成
- 用户明确提到"分步骤做"、"列个计划"、"按 todo 推进"

以下情况无需 Todo：

- 单文件、单步骤的 obvious fix
- 纯问答类咨询（无需执行动作）
- 用户给出了极其具体的一步指令

## Todo 设计原则

### 原则 1：每个 Todo 是一个可独立完成的动作单元

**正确的粒度**：

```
待办事项:
- [pending] 读取当前 skill 内容和相关规则
- [pending] 确定修改范围（3 个文件）
- [pending] 编写修改后的 SKILL.md
- [pending] 运行 validate-skill.sh 验证格式
- [pending] 更新 skills/README.md 索引
```

**错误的粒度**：

```
- [pending] 修改代码              <-- 太大，无法独立验收
- [pending] 修复 bug              <-- 没有边界
- [pending] 读一下然后改一下      <-- 两个动作混在一起
```

### 原则 2：Todo 名称必须是动宾结构

使用"动词 + 名词"的格式，让 AI 和人类都能一眼看出要做什么：

- 正确：`读取后端 API 路由文件`
- 正确：`提取公共逻辑到 util.ts`
- 错误：`代码`、`bug`、`看看`、`处理一下`

### 原则 3：Todo 状态必须明确

只允许四种状态：

| 状态 | 含义 | 使用时机 |
|------|------|----------|
| `pending` | 等待开始 | 初始化时 |
| `in_progress` | 进行中 | 当前正在做的唯一一项 |
| `done` | 已完成 | 该项验收通过后 |

**关键规则**：
- 同一时刻**最多只能有一个** `in_progress` 的 Todo
- 不允许所有 Todo 同时是 `pending`（必须有一个在推进）
- 不允许有已 `done` 的 Todo 后面还跟着 `pending` 时，前面仍有 `in_progress` 卡住不动

### 原则 4：Todo 总数控制在合理范围

- 简单任务：3~5 项
- 中等任务：5~8 项
- 复杂任务：8~12 项，若超过则考虑拆分子任务或 Task Session

如果 Todo 超过 12 项，说明任务粒度仍然太粗或范围太大，需要重新拆分。

## Todo 执行约束（核心规则）

### 规则 1：静默执行，完成全部后再汇报

**这是本 skill 的最高优先级规则。**

一旦建立了 Todo 清单，AI 的行为模式必须切换为：

1. **读取 Todo** → 确认当前 `in_progress` 项
2. **执行当前项** → 完成该动作单元的全部工作
3. **更新状态** → 将当前项标记为 `done`，下一项标记为 `in_progress`
4. **重复 1~3** → 直到所有 Todo 变为 `done`
5. **最终汇报** → 在所有 Todo 完成后，一次性向用户汇报结果

**禁止的行为**：
- 完成第一个 Todo 后就向用户汇报"我已经完成了第一步"
- 每完成一个文件修改就发一条消息
- 在没有明确阻塞的情况下，以"进度更新"为由打断用户

### 规则 2：什么情况下可以中途汇报

只有在以下情况，才允许在未完成全部 Todo 时向用户汇报：

1. **遇到不可逾越的阻塞**
   - 需要的文件不存在且无法推断位置
   - 命令执行失败且无法自行修复
   - 发现任务范围与用户描述严重不符
   - 需要用户在多个互斥方案中做选择

2. **用户主动追问**
   - 用户发消息询问"进度如何"
   - 用户要求"先汇报一下"

3. **系统强制中断**
   - 上下文窗口即将耗尽，必须做 handoff
   - 工具调用次数达到上限

**即使中途汇报，也必须附带完整的 Todo 状态快照**，让用户清楚知道哪些已完成、哪些待处理、当前阻塞在哪里。

### 规则 3：更新 Todo 必须伴随真实进展

**禁止的行为**：
- 连续两次调用 SetTodoList 但没有任何实际的代码/文件/命令操作发生在两次之间
- 只是为了"好看"而提前把 Todo 标为 `done`
- Todo 状态和实际工作不同步

**正确做法**：
- 先执行实际动作（读文件、写代码、跑测试）
- 动作有结果后，再更新 Todo 状态
- 如果某个 Todo 执行失败，将其保留为 `in_progress` 或回退为 `pending`，并记录原因

### 规则 4：工具调用之间不插入解释性文字

当 AI 处于 Todo 执行模式时，每次工具调用之间**不应输出多余的解释文字**。工具调用序列本身就是执行过程的体现。

错误示例：
```
好，我现在来读取第一个文件。
<tool call>
文件读完了，让我分析一下。
<tool call>
分析完了，现在我要修改代码了。
<tool call>
```

正确做法：
```
<tool call: SetTodoList>
<tool call: ReadFile>
<tool call: ReadFile>
<tool call: WriteFile>
<tool call: SetTodoList>
```

只有在需要用户做决策、或汇报最终结果时，才输出自然语言总结。

## Todo 与 Task Session 的关系

### 短期任务（1~3 轮交互可完成）

- 仅使用 SetTodoList 跟踪
- 无需创建 Task Session 文档

### 中长期任务（可能跨会话）

- **必须创建 Task Session** 文档（见 `task-session` skill）
- Todo 清单作为当前轮次的执行跟踪
- Task Session 作为跨轮次的决策和上下文沉淀

关系：
- Task Session 回答"这个任务的目标、边界、关键决策是什么"
- Todo 回答"这一轮我要具体做哪些事、做到哪一步了"

## 汇报格式

### 全部完成后的最终汇报

必须包含以下部分：

1. **结果摘要**（1~2 句话）
2. **具体改动清单**（文件、功能点）
3. **验证结果**（测试是否通过、命令输出）
4. **后续建议**（如有）

示例：
```
已完成全部 5 项 Todo，本次修改如下：

1. 新增 `aiasys-frontend-architecture` skill
   - 文件：`.agents/skill-development/aiasys-frontend-architecture/SKILL.md`
   - 内容：描述了手动路由、Context+Hooks、API 层规范等专属约束

2. 新增 `todo-orchestration` skill
   - 文件：`.agents/skill-development/todo-orchestration/SKILL.md`
   - 内容：规范了 Todo 设计原则、静默执行规则、汇报格式

验证：已运行 `validate-skill.sh`，两个 skill 均通过格式校验。

当前状态：两个 skill 均位于开发迭代区，尚未发布到正式使用区。
```

### 中途阻塞汇报

必须包含：

1. **阻塞原因**（清晰说明为什么停下来了）
2. **已完成的 Todo**（让用户知道进度）
3. **待完成的 Todo**
4. **需要用户做什么**（明确的问题或选择）

示例：
```
任务执行到第 3 步时遇到阻塞，需要您确认：

当前进度：
- [done] 读取当前数据库模型
- [done] 确定需要新增的字段
- [in_progress] 编写 migration 脚本
- [pending] 运行 migration 验证

阻塞原因：现有 migration 版本号与 Alembic 历史冲突，发现两个 head（a1b2c3d 和 e4f5g6h）。

需要确认：应该基于哪个 head 继续生成新的 migration？
```

## 常见错误

### 错误 1：过度汇报

```
AI：我已经读取了文件 A。
AI：我已经读取了文件 B。
AI：我现在开始修改文件 C。
AI：文件 C 修改完成了。
```

这是严重违反本 skill 的行为。所有中间步骤应该静默完成，最终只发一条总结消息。

### 错误 2：Todo 与工作不同步

Todo 里写着"[done] 修复测试"，但实际上测试仍然失败。

这是不可接受的。Todo 状态必须是真实工作状态的准确投影。

### 错误 3：先汇报再补 Todo

AI 先向用户说"我改好了"，然后才想起来更新 Todo。

正确顺序：先完成工作、更新 Todo，最后再向用户汇报。

### 错误 4：Todo 粒度太粗导致中间无法跟踪

一个 Todo 项是"实现整个功能"，执行了 20 个 tool call 期间没有任何进度标记。

虽然本 skill 要求静默执行，但 Todo 本身仍然应该有合理的粒度，让 AI 自己也能明确当前处于哪个阶段。

## 快速检查清单

**创建 Todo 时：**
- [ ] 任务是否需要 3 步以上？是则必须建 Todo
- [ ] 每个 Todo 都是动宾结构、可独立完成
- [ ] 总数在 12 项以内
- [ ] 初始状态只有一个 `in_progress`，其余为 `pending`

**执行 Todo 时：**
- [ ] 每次只推进一个 `in_progress` 项
- [ ] 完成一项后立即更新 Todo 状态
- [ ] 中间步骤不向用户发送进度消息
- [ ] 遇到阻塞才中断并汇报

**汇报时：**
- [ ] 是否所有 Todo 都已完成？是则发最终汇报
- [ ] 汇报内容包含结果摘要、改动清单、验证结果
- [ ] 若中途汇报，必须说明阻塞原因和当前进度
- [ ] 任务涉及 skill 的新增、修改、废弃时，确认是否需要执行关联引用检查（见 AGENTS.md "Skill 变更后的关联维护"和 task-session "文档同步"章节）

## 文档同步提醒

当任务涉及以下操作时，必须在 Todo 中加入对应的同步检查项：

- **修改了某个 skill 的 SKILL.md**（包括 description、职责范围、关联引用）→ 搜索该 skill 名称在 `.agents/skills/` 下的所有引用，确认其他 skill 的"不替代 XXX""详见 XXX"等声明是否需要同步更新
- **新增 skill 到 `.agents/skills/`** → 确认新 skill 的 description 是否声明了与已有 skill 的边界（"不替代 XXX""不适用于 XXX"）
- **废弃或删除 skill** → 搜索该 skill 名称在所有 skill 和 AGENTS.md 中的引用，更新或删除失效引用
- **修改了 AGENTS.md 中的约束规则**（项目语义、协作规则、多终端 AI 协作开发约束等）→ 检查受影响的 skill 是否需要同步更新，确保 skill 中的规则描述不与 AGENTS.md 冲突

这不是可选步骤。skill 之间的引用不一致会导致后续 AI 读取到矛盾或失效信息，产生不可预期的行为。具体协议见 AGENTS.md "Skill 变更后的关联维护"。

---

注意：本 skill 的核心价值是"减少无效交互，提升执行完整性"。AI 应该像一个能独立完成任务的专业工程师，在完成任务后敲门汇报，而不是每做一步就抬头看用户一眼。

## 任务优先级与必要性标记

Todo 清单中的每项任务应标明优先级和必要性，用于指导执行顺序和取舍决策。

**优先级**：

| 标记 | 含义 |
|------|------|
| P0 | 阻塞项，必须最先完成 |
| P1 | 高优先级，本轮必须完成 |
| P2 | 中等优先级，本轮尽量完成 |
| P3 | 低优先级，可推迟到后续轮次 |

**必要性**：

| 标记 | 含义 |
|------|------|
| MUST | 必须实现 |
| SHOULD | 应该实现，但有合理理由可跳过 |
| COULD | 可以实现，但非必要 |
| WONT | 明确不做，仅登记备忘 |

`WONT` 或 `dropped` 标记的任务默认不得进入实现流程。

示例：

```
待办事项:
- [pending] [P0/MUST] 修复登录接口 500 错误
- [pending] [P1/MUST] 新增用户权限校验中间件
- [pending] [P2/SHOULD] 优化数据库查询性能
- [pending] [P3/COULD] 补充 API 文档示例
```

## 过期与废弃任务管理

### 显式下线规则

对已过期、已下线或与当前架构冲突的任务，不得直接删除，必须标记为 `dropped` 并记录原因：

- 下线原因（为什么不再需要）
- 重开条件（什么情况下需要重新考虑）

示例：

```
- [dropped] [P2/SHOULD] WebSocket 实时推送
  下线原因：当前阶段 SSE 已满足需求，WebSocket 引入额外运维复杂度
  重开条件：用户量 > 1000 或需要双向实时通信时重新评估
```

### 每轮对齐补录

完成一轮 Todo 执行后，对照代码与 changelog，补录"已实现但未在需求台账或完成记录中登记"的任务。若当前轮必须追溯旧版计划，可参考 `archive/todo-lists/` 中的历史文件；当前任务默认不再回写 repo 内 todo。

## 发布前双向核对

当任务涉及发布或版本交付时，必须执行双向核对：

**需求/状态 → Changelog**：已完成的事项应能在对应版本 changelog 中找到记录。

**Changelog → 需求/状态**：Changelog 中的新增/修复项应能在需求台账、实现状态或当前 AI 任务记录中找到来源。

核对不通过时，补全缺失的记录后再继续。

## 任务最小字段约束

每条 Todo 项至少应具备以下字段，缺一不可：

| 字段 | 说明 | 必填时机 |
|------|------|----------|
| 优先级 | P0/P1/P2/P3 | 创建时 |
| 必要性 | MUST/SHOULD/COULD/WONT | 创建时 |
| 状态 | pending/in_progress/done/dropped | 始终 |
| 做什么 | 动宾结构描述 | 创建时 |
| 完成标准 | 如何判断该项已完成 | 创建时 |
| 证据 | 验证结果路径或截图 | done 时 |

`dropped` 项额外需要：下线原因、重开条件。

## 执行报告模板

### 全部完成后的最终汇报

```markdown
## 任务情况
- **任务**: [简要描述]
- **已完成内容**: [关键已完成项]

## 是否已经完成
- **状态**: 成功
- **验收状态**: 全部完成

## 生成/修改文件
| 文件 | 路径 | 说明 |
|-----|------|------|
| [类型] | [路径] | [说明] |

## 检查清单
- [x] 检查项1
- [x] 检查项2

## 验证结果
[测试输出、截图路径、构建结果]

## 后续建议
[如有]
```

### 部分完成或阻塞汇报

```markdown
## 任务情况
- **任务**: [简要描述]
- **已完成内容**: [已完成的 Todo 项]

## 是否已经完成
- **状态**: 部分成功 / 失败
- **验收状态**: 部分完成

## 生成/修改文件
| 文件 | 路径 | 说明 |
|-----|------|------|
| [类型] | [路径] | [说明] |

## 未完成项
- [ ] [未完成内容 + 阻塞原因 + 下一步]

## 下一步准备做什么
- [下一步动作]

## 已知限制
- [限制说明]
```

### 状态判定规则

- `成功`：仅当验收项全部完成，且无 TODO/FIXME/占位逻辑残留。
- `部分成功`：存在未完成项或前置条件阻塞，必须明确剩余工作。
- `失败`：目标未达成或关键验证失败。
- 禁止将"已写 TODO/占位"描述为"已完成"。

## 代码修改与命令执行报告

当 Todo 涉及代码修改或命令执行时，使用以下格式记录具体变更：

### 代码修改报告

```markdown
## 修改文件：[文件名]

**修改类型**: [新增/修改/删除]

**修改内容**:
```diff
[diff 内容]
```

**修改原因**:
[说明]

**影响范围**:
[说明]
```

### 命令执行报告

```markdown
## 执行命令
```bash
$ [命令]
```

## 执行结果
```
[输出]
```

## 结果分析
- [状态说明]
```
