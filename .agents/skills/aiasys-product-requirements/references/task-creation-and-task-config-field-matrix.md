# 新建任务与任务配置字段矩阵

状态: 主线专题
更新时间: 2026-04-22
适用版本: v0.3.9 当前主线

---

## 1. 文档目标

本文档用于把“新建任务”和“任务配置”两类入口的字段边界彻底写清楚。

它回答的是：

- 哪些字段属于新建任务阶段
- 哪些字段属于工作区设置
- 哪些字段属于当前分支设置
- 当前代码里哪些字段已经有稳定入口
- 哪些字段修改后更接近立即生效，哪些字段通常在下一次执行生效

---

## 2. 核心结论

### 2.1 新建任务只保留启动必需项

当前代码里的新建任务入口已经进一步收窄。

当前主界面真实保留的只有：

- 任务名称
- 任务说明

当前主界面真实固定展示但不提供编辑的只有：

- 默认执行方式 = 本地 Python

### 2.2 工作区设置与当前分支设置分开承接

- 工作区设置承接长期工作区信息、共享能力、资源挂载与自动化
- 当前分支设置承接当前分支模型覆盖、专家启用策略、分支私有记忆和分支维护动作
- 用户默认层在当前主线里仍然分散在偏好、模型配置和工作区设置内部 defaults scope 预留实现中

### 2.3 当前运行中允许保存，不默认热更新

`active` 状态下，大部分关键字段允许保存，但默认只对下一次执行生效。

## 4. 设计解释

### 4.1 为什么新建任务字段被收得更小

当前主界面的目标是尽快进入一个可用工作区，不再让用户在创建前先做大量技术决策。

因此当前代码里：

- 创建阶段只让用户填写任务名称和说明
- 默认执行方式固定展示为本地 Python
- mode、环境、初始化方式都没有继续暴露给用户

### 4.2 为什么资源挂载继续放在工作区设置

资源挂载本质上是工作区共享事实源，更适合长期绑定在工作区上，而不是被每条分支都复制一份。

### 4.3 为什么分支设置不再承担整套任务配置

当前代码里的分支设置更像“分支私有覆盖和维护面板”，而不是完整配置中心。

它主要负责：

- 当前分支模型覆盖
- 当前分支专家启用策略
- 当前分支私有记忆
- 历史压缩和分支记录查看

运行态重置、资源挂载、工作区能力目录这些内容仍然落在别的入口。

### 4.4 为什么资源挂载在当前工作过程中也允许改

因为很多任务会在执行过程中逐步补材料。

但是否立即对当前正在运行的一轮执行可见，应由 runtime 刷新策略决定，而不是在产品文档里假定为热更新。

---

## 5. 当前建议结论

当前主线应统一以下规则：

1. 新建任务只负责轻量启动。
2. 工作区设置与当前分支设置必须拆开。
3. 当前主界面的字段矩阵应以“已实现入口”为准，不再把旧 mode / 环境创建表单当成主线。
4. 前端必须明确告诉用户哪些修改是立即生效，哪些是下次执行生效。
