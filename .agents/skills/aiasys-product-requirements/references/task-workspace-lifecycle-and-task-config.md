# 任务工作区生命周期与任务配置产品设计

状态: 主线专题
更新时间: 2026-04-22
适用版本: v0.3.9 当前主线

---

## 1. 文档目标

本文档用于收敛三个容易混淆的产品问题：

1. 单个任务工作区的生命周期是什么
2. 新建任务时到底需要用户先决定什么
3. 进入任务后，任务配置负责什么

本文档属于产品语义文档，不替代具体接口字段和状态机实现。

当前主线补充约束：

- 不继续为旧 session-first 语义设计长期兼容迁移
- 可以在进入新模型前清理本地真实运行历史
- 后续接口与交互动作优先围绕 `任务工作区 / session / 执行轮次` 展开

当前主线术语约定：

- 本稿统一使用 `session（当前分支）`
- 代码、接口或旧界面文案里仍然存在的 `conversation / 对话 / conversation_id / current_conversation`，当前都按兼容别名理解

## 2. 任务工作区与 session 的生命周期关系

### 2.1 对象层级

当前主线应把三层对象区分开：

- 任务工作区
- 工作区下的 session
- session 中的执行轮次

其中：

- 任务工作区是长期对象
- session 是工作区内部的记忆分支
- 执行轮次是某次具体运行的证据单元

### 2.2 当前代码中的工作区主状态

当前后端已经给工作区保留了 `status` 字段，但现有主线并没有实现完整的工作区生命周期状态机。

更接近代码真相的口径是：

- 创建工作区时默认写入 `status: active`
- 删除工作区时直接执行删除，不经过显式 `deleting / deleted` 中间态
- `completed` 目前主要还是分支执行和研究自动化里的语义，不是工作区主状态

因此这里应明确区分：

- 工作区状态字段已经存在
- 但工作区主生命周期还没有在当前代码里细化成稳定状态机
- 当前代码真正细分较多的是“分支执行状态”和“运行态状态”

### 2.3 session 生命周期默认独立于工作区生命周期

session 不应复用工作区的主生命周期。

session 更适合作为工作区内部的可切换、可新增、可归档对象，例如：

- 当前 session
- 历史 session
- 新建空 session
- 已归档 session

也就是说：

- 结束一个 session 不等于结束整个任务工作区
- 清空记忆不等于删除整个任务工作区
- 新建任务也不等于只是在当前工作区里开个新 session
- Fork session 也不等于新建任务，它只是当前工作区中的分支 session

当前代码还需要额外说明一层事实：

- 后端已经有 `workspace -> session 投影` 这层结构
- 但 `conversation_id` 目前仍直接复用 `session_id`
- 删除、切换、运行态刷新等很多链路仍然直接围绕 `session_id` 驱动

所以“session 层已经完全摆脱旧 `conversation` 兼容字段”并不是当前真相。

### 2.4 执行状态与任务主状态分离

任务工作区的主状态必须与执行状态分离。

执行状态用于表达当前一轮是否正在运行，例如：

- `idle`
- `running`
- `waiting_user`
- `succeeded`
- `failed`
- `cancelled`

执行失败不应直接把任务工作区打成“永久失败”。

### 2.5 运行态状态与任务主状态分离

任务工作区的主状态也必须与运行态状态分离。

运行态状态用于表达当前 runtime 是否可立即执行，例如：

- `not_initialized`
- `ready`
- `busy`
- `refresh_required`
- `missing`

这层状态回答的是“当前能不能直接跑”，不是“这个任务是否结束”。

### 2.6 `completed` 在当前代码里更接近执行侧语义

当前代码里的 `completed` 更接近：

- 某条分支上一轮执行已经结束
- 或某类自动化 / 研究流程节点已经结束

它当前不应被解释成“工作区已经完成并关闭”。

## 3. 新建任务与任务配置的边界

### 3.1 当前代码里新建任务只收集最小信息

当前主线的新建任务弹窗已经进一步收窄。

当前实际收集的字段只有：

- 任务名称（必填）
- 任务说明（可选）

当前实际固定展示但不让用户编辑的是：

- 默认执行方式 = 本地 Python

也就是说，当前主界面里的“新建任务”并不会在创建阶段让用户选择：

- mode
- 初始运行环境
- 初始化方式

当前代码里的真实创建链路是：

- 先准备一个新的 session 草稿
- 再创建新的任务工作区
- 自动创建首个默认 session
- 最后进入这个默认分支开始工作

当前推荐的动作命名是：

- `新建任务`
- `新建 session`
- `清空记忆并新建 session`
- `Fork session`

不建议把 `新建工作区` 与 `新建任务` 做成两个并列主按钮。

仓库里仍保留少量 `新建对话`、`Fork 对话` 这类旧文案和旧代码残留，但它们不再作为当前主线术语。

### 3.2 新建任务不应变成大表单

以下内容不应作为新建任务的默认必填项：

- 完整任务级 Agent 配置
- 完整 MCP 列表
- Skill 包安装与卸载
- 详细资源挂载
- 大量高级策略开关

原因很简单：

- 创建流程会过重
- 生命周期边界会模糊
- 用户往往进入任务后才知道要不要改

### 3.3 工作区设置与当前分支设置必须分开

任务创建成功后，后续配置入口不能继续只叫一个笼统的“工作区配置”。

更稳定的模型是：

- `工作区设置`
  - 负责长期工作区信息与共享资源
- `当前分支设置`
  - 负责当前分支这次实际如何执行

两者需要回答的问题不同：

- 工作区设置回答：
  - 这个工作区叫什么
  - 这个工作区默认执行方式摘要是什么
  - 这个工作区共享挂载了哪些资源
- 当前分支设置回答：
  - 当前分支是否覆盖了模型
  - 当前分支启用了哪些专家角色
  - 当前分支私有记忆是否待生效
  - 当前分支有哪些维护动作入口

当前代码里已经落地的边界更接近下面这组职责：

- 工作区设置
  - 工作区标题与描述
  - 工作区默认执行方式摘要
  - Skill 挂载
  - 工作区专家目录
  - 能力目录与运行时可用性摘要
  - 数据库 / 知识库 / 知识图谱挂载
  - 自动化与定时任务
  - 工作区共享记忆
- 当前分支设置
  - 当前分支模型覆盖
  - 当前分支专家启用策略
  - 当前分支私有记忆
  - 分支记录查看入口
  - 历史压缩入口
  - 当前分支运行态摘要

当前还没有完全收口进“当前分支设置”的动作包括：

- 当前分支运行态重置
- 更细粒度的工具切换
- mode 切换

## 4. 工作区设置与当前分支设置的产品范围

进入任务工作区后，至少需要区分两类配置对象：

- 工作区设置
- 当前分支设置

### 4.1 工作区设置

- 工作区标题
- 工作区描述
- 工作区默认执行方式摘要
- 工作区共享资源挂载
- 工作区共享能力摘要
- Skill 挂载
- 工作区专家目录
- 自动化与定时任务
- 工作区共享记忆

### 4.2 当前分支设置

- 当前分支模型覆盖
- 当前分支专家启用策略
- 当前分支私有记忆
- 历史压缩
- 分支记录查看入口
- 当前分支运行态摘要

当前代码里仍未完全放进这个弹窗的动作：

- 当前分支运行态重置
- 当前分支 mode 切换
- 更细粒度的工具启用切换

### 4.3 session 层默认负责什么

session 层当前默认负责：

- 当前记忆上下文
- 当前推进路径
- 当前 session 历史
- 与该 session 关联的执行证据入口
- 从现有 session 内容 Fork 出新的分支 session

session 层当前默认不负责：

- 独立复制一套完整任务配置
- 单独复制一份工作区文件树
- 重定义任务工作区本身的生命周期

但 session 层需要允许：

- 保存当前分支级 Agent 覆盖
- 保留当前分支兼容预置和执行事实
- 在当前分支已经有聊天或执行记录时锁定 mode，改为新建另一条分支

## 5. 当前代码与目标态差距

当前代码还没有围绕工作区实现完整的 `draft / active / completed` 生命周期状态机。

更接近当前主线的事实是：

- 工作区元数据默认只有 `active`
- 当前分支是否“空白 / 已开始 / 已有执行记录”主要通过 `message_count`、`execution_record_count` 和历史记录判断
- 本轮是否运行中主要通过 `isRunning`、`runtime_summary` 和执行记录来表达
- 配置是否待生效主要通过 `pending_*_version`、`applied_*_version` 和 `memory_effect` 这类字段表达

因此后续如果要继续写工作区生命周期文档，必须把“工作区状态”“分支状态”“运行态状态”“执行证据状态”分开，不要再混成一组。

### 5.1 同一工作区下新建 session

- 默认复用当前工作区已有任务级配置
- 不因新 session 而清空工作区文件、产物和资源挂载
- 当前 session 的记忆与历史从空开始

### 5.2 同一工作区下 Fork session

- 默认复用当前工作区已有任务级配置
- 保留原 session 不变
- 新 session 继承来源 session 已有历史
- 当前代码还会一起克隆数据库附着这类分支级资源
- 新 session 后续的执行记录独立沉淀

---

## 6. 当前产品结论

当前主线必须坚持以下结论：

1. 单个任务工作区必须明确分层：任务主状态、执行状态、运行态状态。
2. 单个任务工作区下允许存在多个 session，session 生命周期独立于工作区主生命周期。
3. 新建任务 = 新建工作区并创建首个默认 session，不等于仅新开一条 session。
4. 清空记忆重新开始默认优先实现为在当前工作区中创建新的空 session，而不是删除工作区资产。
5. Fork session 属于当前工作区中的分支能力，不等于新建任务。
6. 当前任务相关配置已经拆成“工作区设置 + 当前分支设置”，不是单一总入口，也不是每个 session 都单独复制一套完整配置。
7. 当前运行中允许保存配置，但默认不承诺热更新。
