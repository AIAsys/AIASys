# 废弃与迁移

**代码是负债，不是资产。**

每行代码都有持续的维护成本：修复 bug、更新依赖、安全补丁、新人上手。废弃是移除不再值得的代码的纪律，迁移是安全地把用户从旧系统移到新系统的过程。

大多数组织擅长构建，却很少擅长拆除。这个 skill 填补这个空白。

**关联文档：**
- `code-simplification`：废弃后的代码清理和简化
- `frontend-pattern`：涉及前端组件废弃时的架构约束
- `api-dev`：涉及 API 废弃时的路由和模型迁移
- `bug-discovery`：废弃过程中发现的 bug 提报流程

---

## 使用场景

- 用新系统、API 或库替换旧的
- 下线不再需要的功能
- 整合重复实现
- 移除无人维护但被依赖的"僵尸代码"
- 设计新系统时就规划其生命周期

---

## 核心原则

### 代码是负债

代码的价值在于它提供的功能，不在于代码本身。当同样的功能能用更少、更简单的代码提供时，旧代码就应该被移除。

### Hyrum's Law 让移除变难

当用户足够多时，每一个可观察行为都会被依赖——包括 bug、时序巧合、未记录的副作用。因此废弃需要**主动迁移**，而不是只发公告。

### 废弃规划从设计时开始

构建新系统时就要问："三年后我们如何移除它？" 接口干净、功能开关完善、暴露面小的系统更容易废弃。

---

## 废弃决策

动手前回答这些问题：

1. **这个系统还提供独特价值吗？**
   - 是 -> 继续维护
   - 否 -> 继续评估

2. **有多少用户/消费者依赖它？**
   - 量化迁移范围

3. **有替代方案吗？**
   - 没有 -> 先建替代方案，不要无替代废弃

4. **每个消费者的迁移成本多高？**
   - 可自动化 -> 直接做
   - 高成本手工 -> 权衡维护成本 vs 迁移成本

5. **不废弃的维护成本有多高？**
   - 安全风险、工程师时间、复杂度机会成本

---

## 强制 vs 建议废弃

| 类型 | 使用时机 | 机制 |
|------|----------|------|
| **建议型** | 迁移可选，旧系统稳定 | 文档、警告、提示，用户自行安排 |
| **强制型** | 旧系统有安全漏洞、阻塞进展、维护成本不可持续 | 硬截止日期，提供迁移工具 |

**默认建议型。** 只有在维护成本或风险足够高时才使用强制型。强制废弃必须配套提供迁移工具、文档和支持。

## 迁移流程

### 步骤 1：建好替代方案

不要在没有可用替代的情况下废弃。替代方案必须：

- 覆盖旧系统的所有关键用例
- 有文档和迁移指南
- 已在生产环境验证（不只是理论上更好）

### 步骤 2：公告与文档

```markdown
## 废弃通知: OldService

**状态:** 自 2025-03-01 起废弃
**替代方案:** NewService（见下方迁移指南）
**移除日期:** 建议型 — 暂无硬截止
**原因:** OldService 需要手动扩容且缺乏可观测性，
        NewService 自动处理两者。

### 迁移指南
1. 将 `from old_service import client` 替换为 `from new_service import client`
2. 更新配置（见下方示例）
3. 运行验证脚本: `python -m scripts.migrate_check`
```

### 步骤 3：增量迁移

逐个迁移消费者，不要一次性全部切换。对每个消费者：

1. 识别所有与旧系统的接触点
2. 更新为使用替代方案
3. 验证行为一致（测试、集成检查）
4. 移除对旧系统的引用
5. 确认无回归

**迁移责任原则：** 如果你拥有被废弃的基础设施，你有责任迁移你的用户，或提供向后兼容的更新。不能只发公告就不管。

### 步骤 4：移除旧系统

确认所有消费者都已迁移后：

1. 验证零活跃使用（通过指标、日志、依赖分析）
2. 移除代码
3. 移除相关测试、文档、配置
4. 移除废弃通知
5. 庆祝——移除代码是成就

## 迁移模式

### 绞杀者模式（Strangler Pattern）

新旧系统并行运行，逐步将流量从旧系统切到新系统。当旧系统流量为 0% 时移除。

```
阶段 1: 新系统 0%，旧系统 100%
阶段 2: 新系统 10%（灰度）
阶段 3: 新系统 50%
阶段 4: 新系统 100%，旧系统空闲
阶段 5: 移除旧系统
```

### 适配器模式

创建适配器，将旧接口翻译到新实现。消费者在迁移期间继续使用旧接口。

```python
class LegacyTaskAdapter:
    def __init__(self, new_service: NewTaskService):
        self._new = new_service

    def get_task(self, id: int) -> OldTask:
        task = self._new.find_by_id(str(id))
        return self._to_old_format(task)
```

### 功能开关迁移

使用功能开关逐个切换消费者：

```python
def get_task_service(user_id: str) -> TaskService:
    if feature_flags.is_enabled("new-task-service", user_id=user_id):
        return NewTaskService()
    return LegacyTaskService()
```

## 僵尸代码

**特征：**

- 6 个月以上无提交，但有活跃消费者
- 没有明确的维护者或负责团队
- 测试失败但无人修复
- 依赖有已知漏洞但无人更新
- 文档引用已不存在的系统

**处理方式：** 要么指定 owner 并正常维护，要么制定具体迁移计划并废弃。不能让它一直悬置。

---

## 常见借口与反驳

| 借口 | 反驳 |
|------|------|
| "它还能跑，为什么要移除？" | 无人维护的代码会默默积累安全债务和复杂度 |
| "以后可能还用得上" | 真需要时可以重建，保留无用代码的成本高于重建 |
| "迁移太贵了" | 对比 2-3 年的维护成本，迁移通常更便宜 |
| "等新系统做完再废弃" | 废弃规划应在设计时就做，做完后会有新优先级 |
| "用户会自己迁移的" | 他们不会。提供工具、文档和激励，或自己做迁移 |
| "我们可以两边都维护" | 两套系统 = 双倍的维护、测试、文档和上手成本 |

---

## 红旗信号

- 没有替代方案就宣布废弃
- 只有废弃通知，没有迁移工具或文档
- "建议型"废弃多年没有任何进展
- 僵尸代码有消费者但无 owner
- 向已废弃系统添加新功能
- 未测量当前使用量就移除代码
- 未验证零消费者就删除系统

## 验证清单

完成废弃后检查：

- [ ] 替代方案已在生产验证，覆盖所有关键用例
- [ ] 存在包含具体步骤和示例的迁移指南
- [ ] 所有活跃消费者已完成迁移（通过指标/日志验证）
- [ ] 旧代码、测试、文档、配置已完全移除
- [ ] 代码库中无任何对已废弃系统的引用
- [ ] 废弃通知已移除（已完成使命）

## 残留清理

旧系统移除后，往往留下死代码、过时文件、废弃引用。以下决策矩阵指导清理工作。

**核心原则：删除是不可逆操作。宁可多问一次，不要误删一个。**

### 可以直接删除（无需询问）

| 类型 | 判断条件 | 示例 |
|------|----------|------|
| 已 gitignore 的构建产物 | `git check-ignore <path>` 返回成功 | `dist/`、`node_modules/`、`__pycache__/`、`.vite/` |
| 自己本次改动产生的废弃 import/变量 | 仅限当前修改范围内 | 重构后残留的旧 import |
| 明确的临时脚本且用完即弃 | 文件名含 `test_`、`tmp_` 且 gitignore | `scripts/test_foo.py`（用完即归档或删除） |

### 必须汇报后等你确认

| 类型 | 原因 |
|------|------|
| **任何目录的删除** | 目录删除影响面大，即使 grep 无引用也可能有运行时依赖、构建配置引用、动态路径拼接 |
| **非代码文件**（`.md`、`.json`、`.yaml`、`.png` 等） | 引用方式多样（文档链接、构建脚本、运行时读取），grep 不可靠 |
| **页面/路由级目录** | 即使无路由入口，内部组件可能被其他页面 import |
| **配置文件** | 可能被环境变量、部署脚本、CI 间接引用 |
| **含多个文件的模块目录** | 实现完整但"暂时未接入"可能是半成品，不是遗留代码 |

### 绝对不要删除

| 类型 | 判断特征 |
|------|----------|
| "来不及集成"的半成品 | 实现完整、属于产品功能链的自然环节、只是暂时未接入调用方 |
| 有其他文件 import 的文件 | grep 能找到跨文件引用 |
| 用户明确要求保留的 | 不要二次判断 |

### 清理流程

1. 用 `Shell` + `Grep` 扫描目标目录，列出所有可疑项，按上述矩阵分类
2. 向用户汇报：确认可删（附判断依据）、疑似可删但需要确认（说明为什么不确定）、不应删除（说明原因）
3. 用户确认后执行删除，完成后验证构建/测试通过

### 判断"遗留代码"的标准

真正的遗留代码特征：

1. 有明确的废弃标记或注释（`@deprecated`、`TODO: remove` 等）
2. 属于已被替换的旧实现，且调用方已迁移
3. 有代码质量问题（冗余逻辑、永远为真的条件）

**以下不是遗留代码**：

- 实现完整但暂时未接入调用方的模块
- 属于产品功能链自然环节的半成品
- 仅在当前分支无引用，但其他分支或 pending PR 中使用的

### 常见陷阱

- **grep 搜不到引用不等于死代码**：动态 import、字符串拼接路径、构建配置中的引用、文档中的链接，grep 都搜不到
- **目录名暗示旧功能不等于可删**：比如 `Knowledge/` 目录虽然没有路由，但组件被其他地方 import
- **gitignore 的文件不全是可删的**：只有构建产物和缓存可以，手动创建的配置模板（如 `.example` 文件）虽然 gitignored 也不应删除
- **"看起来没用"不是删除理由**：必须用工具验证引用关系

## 拆分后遗留副本（Split-Leftover Pattern）

这是 AIASys 项目中最常见的一类死代码：一个模块被拆分成多个子模块后，原模块中留下了未被删除的函数副本。

### 识别方法

当一个文件既是"聚合层"（通过 `include_router` 注册子模块）又包含私有函数定义时，检查这些私有函数是否在其他子模块中有等价副本。

**三步确认法：**

1. **找副本**：对每个私有函数，在子模块中搜索同名函数
2. **比内容**：逐行对比两个版本是否一致（或仅有格式差异）
3. **查调用**：确认原文件中的版本没有被任何调用方引用（文件内部不调用、外部不导入），而子模块中的版本有活跃调用方

如果三步都满足，则原文件中的版本是遗留副本，可以安全删除。

### 删除流程

1. **归档**：将待删除代码完整备份到 `design-draft/archive/legacy-code-removal/<date>-<module-name>/`，同时写一份 `RECORD.md` 记录每个函数的活跃版本位置和调用方
2. **删除**：移除函数定义和因删除而不再需要的 import
3. **验证**：确保模块正常导入、应用正常启动、相关路由正常响应

### 归档目录结构

```
design-draft/archive/legacy-code-removal/
└── 2026-05-27-sessions-dead-functions/
    ├── RECORD.md          # 每个函数的判定依据和活跃版本位置
    └── original_code.py   # 原始代码完整备份
```

`RECORD.md` 至少包含：每个函数名、原文件行号、活跃版本所在文件及行号、调用方位置、两个版本的差异说明、恢复方法。

### 不是拆分遗留副本的情况

- 子模块中没有同名函数 → 可能是独立实现，需进一步判断
- 原文件中的版本有外部 import → 说明仍在使用，不能删
- 两个版本逻辑不同 → 可能是独立演化，需逐个判断
