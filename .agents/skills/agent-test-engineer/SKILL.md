# Agent 测试工程

AIASys Agent 系统的端到端能力测试。测试用例用接近真实用户输入的问题驱动 Agent 执行，按验收标准判断通过与否，把发现的缺陷反馈到工具设计和提示词改进中。

单靠单元测试无法覆盖"Agent 理解用户意图 -> 选择正确工具 -> 以正确路径执行"这条完整链路，这些测试用例补上这个缺口。

## 触发条件

- 用户要求新增测试用例
- 用户要求运行 Agent 能力测试
- 用户要求批量验证 Agent 工具
- 发现 Agent 行为异常，需要沉淀为回归用例
- 修改工具定义或提示词后，需要跑回归
- 需要建立"测试 -> 观察 -> 分析 -> 修复 -> 重测"的迭代闭环

## 核心方法论：观察驱动的迭代优化

本 skill 的核心不是一次性跑完测试，而是通过系统化的观察和记录，形成可迭代的优化闭环。

```
编写用例 -> 执行测试 -> 观察记录 -> 根因分析 -> 修复优化 -> 重测验证 -> 更新记录
     ^                                                            |
     |------------------------------------------------------------|
```

### 观察什么

每次测试必须记录以下维度：

| 观察维度 | 记录内容 | 用途 |
|---------|---------|------|
| 工具调用链 | Agent 调用了哪些工具、调用顺序 | 判断 Agent 是否选择了正确工具 |
| 参数构造 | 每个工具调用的参数值 | 发现参数格式错误（如 StrReplaceFile 的 edit 字段） |
| 工具结果 | 成功/失败、返回内容、is_error | 验证工具本身是否有 bug |
| Agent 文本回复 | 工具调用后是否生成了解释性文本 | 发现"工具失败后静默"问题 |
| 副作用 | 文件是否创建/修改、环境变量是否变化 | 验证实际效果 |
| 时间消耗 | 每次调用的耗时 | 发现性能问题 |

### 记录格式

测试记录写入用例文件的"测试记录"栏：

```markdown
## 测试记录
- YYYY-MM-DD：结果（通过/部分通过/失败/阻塞）+ 关键发现 + 工具调用链摘要
```

关键发现必须包含根因分析，不能只描述现象。例如：

- 不好：`StrReplaceFile 调用失败`
- 好：`StrReplaceFile 调用失败，根因为 edit 参数被模型输出为 JSON 字符串而非嵌套对象，Pydantic model_validate 拒绝解析。需增加 model_validator 前置解析`

## 测试合集位置

用例内容不在本 skill 中维护，统一放在 `design-draft/agent-test-cases/`：

```
design-draft/agent-test-cases/
├── README.md              # 用例索引（全部用例的状态和编号）
├── run_case.py            # 单条用例执行器
├── run_all_cases.py       # 串行批量执行
├── run_all_concurrent.py  # 并发批量执行（推荐）
├── run_failed_cases.py    # 只跑上次失败的用例
├── report.json            # 最近一次批量运行报告（自动生成）
└── cases/                 # 用例目录（每个用例一个文件夹）
    ├── fs-001-read-file/
    │   ├── README.md      # 用例描述、prompt、验收标准、测试记录
    │   └── fixtures/      # 需要复制到工作区的测试文件（可选）
    └── ...
```

**新增用例前先读 `cases/README.md`**，确认归属的功能域和下一个可用编号。

## 用例格式

```markdown
# TC-{编号} {标题}

## 问题
给 Agent 的完整指令

## 前置条件
需要预置的文件或环境状态

## 验收标准
1. 可自动化验证的检查项

## 常见陷阱
- 陷阱 N：描述 + 根因分析

## 测试记录
- YYYY-MM-DD：结果（通过/部分通过/失败/阻塞）+ 关键发现 + 根因分析
```

字段约束：

| 字段 | 约束 |
|------|------|
| 问题 | 可直接发给 Agent 的完整指令，不能包含元指令（"你应该""请确保"） |
| 前置条件 | 需要预置的文件或环境状态，无则写"无需预置文件" |
| 验收标准 | 每条必须可自动化验证，不接受主观判断 |
| 常见陷阱 | 记录根因（工具设计缺陷/提示词不明确/路径解析错误/参数格式问题/工具缺失），用于回归锚点 |
| 测试记录 | 每次实际跑过就追加一行，标注日期、结果、关键发现和根因分析 |

## 编号规则

`{域缩写}-{三位序号}`，从 001 开始：

| 域 | 缩写 | 域 | 缩写 |
|----|------|----|------|
| 文件系统 | fs | Notebook | nb |
| Shell | sh | Skill 管理 | sk |
| 专家 | ex | MCP | mc |
| 工作区管理 | ws | 环境变量 | ev |
| 知识库 | kb | 知识图谱 | kg |
| Canvas | cv | 子 Agent | sa |
| 自动任务 | at | 数据分析 | da |

## 测试执行

### 前置条件

1. 后端服务在 13001 端口运行
2. 配置好的 LLM provider 可用

### 执行方式

**方式一：单条执行（调试用）**

```bash
cd design-draft/agent-test-cases
python3 run_case.py <session_id> cases/fs-001-read-file/
```

调试模式（输出完整工具调用和结果）：

```bash
python3 run_case.py <session_id> cases/fs-001-read-file/ --debug
```

只输出错误：

```bash
python3 run_case.py <session_id> cases/fs-001-read-file/ --errors-only
```

**方式二：并发批量执行（回归用，推荐）**

```bash
cd design-draft/agent-test-cases
python3 run_all_concurrent.py
```

只跑指定 case：

```bash
python3 run_all_concurrent.py fs-001 sa-003
```

只跑上次失败的：

```bash
python3 run_all_concurrent.py --failed-only
```

**方式三：串行批量执行**

```bash
python3 run_all_cases.py
```

**方式四：重测失败用例**

```bash
python3 run_failed_cases.py
```

### 执行注意事项

- **session 隔离**：每个域尽量使用独立 session，避免上下文污染。已执行过测试的 session 会累积工具调用历史，影响后续测试
- **副作用清理**：每批测试后清理创建的文件、环境变量、skill 安装状态
- **前置条件准备**：测试前确认 fixtures 已正确复制（run_case.py 会自动处理）
- **批量执行间隔**：并发批量执行已内置信号量控制，无需手动间隔

## 根因分析框架

测试失败后，按以下框架分析根因：

| 类别 | 判断标准 | 修复方向 |
|------|---------|---------|
| 工具设计缺陷 | 工具被调用但参数验证失败、内部报错 | 修改工具参数 schema 或解析逻辑 |
| 提示词不明确 | 工具存在但 Agent 未调用，或用 Shell 绕过 | 优化系统提示词中的工具使用指引 |
| 工具描述不清 | Agent 调用错误工具，或找不到正确工具 | 优化工具 description，增加关键词 |
| 参数格式问题 | Agent 构造的参数格式不符合 schema | 简化参数结构，或增加前置解析 |
| 工具缺失 | 系统中不存在对应工具类 | 开发新工具并注册到 baseline |
| 路径解析错误 | 工具调用成功但操作了错误路径 | 修正路径解析逻辑 |
| 上下文污染 | 同一 session 中前面的测试影响后面 | 使用独立 session 隔离 |

## 迭代优化闭环

### Step 1: 执行测试并记录

- 发送 prompt
- 收集 SSE 流
- 记录工具调用链、参数、结果、耗时
- 在用例文件中更新测试记录

### Step 2: 根因分析

- 对照验收标准，判断通过/部分通过/失败
- 失败时按根因分析框架定位问题类别
- 补充或更新"常见陷阱"

### Step 3: 修复优化

- 工具设计缺陷 -> 修改工具代码
- 提示词不明确 -> 修改 general_host_prompt.md
- 工具描述不清 -> 修改工具的 description
- 参数格式问题 -> 修改参数 schema 或增加 model_validator
- 工具缺失 -> 开发新工具并注册到 baseline

### Step 4: 重测验证

- 使用干净 session 重新执行失败的用例
- 确认修复生效
- 更新测试记录

### Step 5: 回归测试

- 修复可能影响其他用例时，跑相关域的全部用例
- 确保没有引入新的回归问题

## 辅助脚本

所有脚本统一放在 `design-draft/agent-test-cases/` 下：

| 脚本 | 用途 |
|------|------|
| run_case.py | 单条用例执行，支持 --debug 和 --errors-only |
| run_all_cases.py | 串行批量执行 |
| run_all_concurrent.py | 并发批量执行，支持过滤和重试 |
| run_failed_cases.py | 只跑上次 report.json 中失败的用例 |

## 边界规则

- 用例文件不超过 30 个用例，接近上限时按子域拆分
- 发现 Agent 异常时先确认是否是已知陷阱，新问题补充陷阱并关联 bug 跟踪
- 测试记录必须包含根因分析，不能只写"失败"或"通过"
- 迭代优化时，每次修改后必须重测验证，不能只改代码不测
- 本 skill 不维护用例内容本身，只维护方法论、流程、格式、编号规则。用例内容在 `design-draft/agent-test-cases/cases/`
