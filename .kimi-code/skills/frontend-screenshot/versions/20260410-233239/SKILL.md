# frontend-screenshot

## 使用场景

- 为根 README 生成或更新展示图
- 为 `docs/` 教程补步骤截图
- 为功能评审保留当前 UI 证据图
- 为前端改动做“真实页面 -> 图片文件”的稳定导出

## 不适用场景

- 只是想看当前页面，不需要落图片文件
- 只是做设计稿，不是从真实前端页面抓图
- 需要 AI 生成插画、海报、贴图，这类应走图片生成而不是截图

## 核心规则

### 规则 1：先截真实前端，不要截代码想象图

这个 skill 只处理“真实运行中的前端页面”。

因此先确认：

- 前端服务已经启动
- 要截图的路由能真实打开
- 页面关键内容已经加载完成

在 AIASys 本仓库里，默认前端地址通常是：

```text
http://127.0.0.1:13000
```

### 规则 2：优先用 `browser-cdp` 的稳定浏览器基线

本仓库/本主机截图默认依赖：

- `browser-cdp`
- `chrome-devtools`

默认视口按笔记本比例处理，而不是手机比例：

```text
1440x900x1
```

如果截图看起来像窄屏手机，先修正视口，再谈页面效果。

### 规则 3：先用快照判断“页面是否就绪”，再真正截图

不要一打开路由就立刻截图。

推荐顺序：

1. 导航到目标页面
2. `take_snapshot` 看结构和关键文案
3. 必要时 `wait_for` 等关键文本出现
4. 再 `take_screenshot`

至少确认：

- 标题出现
- 主要卡片或面板出现
- Skeleton / Loading / 空白占位已经消失
- 没有弹窗、菜单、hover 态遮挡内容

### 规则 4：图片区分“临时证据”和“正式资产”

不要把所有截图都扔到同一个目录。

本仓库建议分两类：

1. 临时调试 / 评审证据
   - 放 `artifacts/`
   - 这是本地过程产物
   - 不作为长期文档资产

2. 需要提交到仓库的正式展示图
   - 放 `images/` 下的明确子目录
   - 例如：
     - `images/readme/`
     - `images/getting-started/`
     - `images/<doc-name>/`

如果用户明确说“不上传 GitHub”，默认先放 `artifacts/`。

### 规则 5：截图前先整理页面状态

真正用于展示的图片，不能带这些噪音：

- 浏览器右键菜单
- hover 态
- tooltip
- 未关闭的 dialog / dropdown
- 开发用 query 参数里造成的异常空态
- 滚动到一半的尴尬位置

必要时先点击、切换 tab、关闭弹层，再截图。

### 规则 6：截图后必须做一次人工式复查

截图成功不等于可用。

至少检查：

- 是否裁切
- 字体是否发虚
- 卡片是否被遮挡
- 间距是否异常
- 是否截到了滚动条、浏览器异常边缘、过多空白
- 文案是否正好停在加载中间态

如果不合格，立即重截，不要把坏图直接留给后续文档使用。

## 执行步骤

### 最短工作流

1. 确认前端页面可打开

2. 进入稳定浏览器基线

推荐先使用：

- `browser-cdp`
- `chrome-devtools`

3. 导航并等待页面稳定

典型顺序：

```text
navigate_page -> emulate(viewport=1440x900x1) -> take_snapshot -> wait_for -> take_screenshot
```

4. 选择输出目录

- 临时证据：`artifacts/<name>.png`
- 正式文档图：`images/<scope>/<name>.png`

5. 截图后立即复查

- 如果是 README / docs 用图，确认画面足够干净和稳定
- 如果是功能证据图，确认关键状态和入口都在画面里

## 常见输出模式

### 模式 1：README 展示图

特点：

- 更关注整体气质
- 尽量选完整页面或关键主视图
- 文件放 `images/readme/`

适合：

- 首页 Hero
- 已打开工作区总览
- 知识图谱主工作台

### 模式 2：文档步骤图

特点：

- 更关注“这一步到底点哪里”
- 画面要稳定、聚焦、少噪音
- 文件放 `images/<doc-name>/`

适合：

- getting-started
- 操作流程文档
- 设置说明文档

### 模式 3：本地评审证据图

特点：

- 只要真实、可复查即可
- 可以保留一点调试语义
- 默认放 `artifacts/`

适合：

- UI 回归前后对比
- 提交前留档
- 设计稿和真实前端对照

## 常见问题

### 问题 1：截图像手机，不像桌面

原因通常是视口不对。

先显式设置：

```text
viewport = 1440x900x1
```

### 问题 2：图里还是 loading / skeleton

不要急着截图。

先：

- `take_snapshot`
- `wait_for` 关键文案
- 必要时稍等一轮再截

### 问题 3：图片该放哪不清楚

按是否需要提交区分：

- 要提交：`images/...`
- 只是本地证据：`artifacts/...`

### 问题 4：一个页面太长

不要默认全页长图。

优先选择：

- 当前首屏
- 关键容器
- 关键面板

只有当用户明确需要完整长图时，才用 full-page screenshot。

## 和其他 Skill 的关系

- `browser-cdp`
  - 负责把浏览器稳定到可截图状态
- `frontend-screenshot`
  - 负责把真实前端页面导出成可用图片文件
- `pencil-render-review`
  - 负责看 `.pen` 的渲染效果，不负责真实前端页面截图
- `frontend-design`
  - 负责设计和视觉质量，不负责截图流程本身

## 交付检查

完成一次“前端展示效果转图片”，至少满足：

- 目标页面来自真实前端，而不是设计稿或静态想象
- 截图前已确认页面稳定
- 图片已落到明确目录
- 如果是正式展示图，文件命名和目录语义清楚
- 已复查图片质量，没有明显遮挡、裁切和加载中状态
