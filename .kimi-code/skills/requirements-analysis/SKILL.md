# 需求分析

将模糊的用户意图转化为明确的开发规格，确保目标对齐、范围清晰、验收标准可验证。

---

## 何时使用

**必须使用：**
- 用户需求模糊或不完整
- 涉及多个功能点的复杂任务
- 需要技术方案选型的任务
- 用户说"做个 XXX 功能"但没有细节

**建议使用：**
- 任何非 trivial 的任务开始前
- 需要估算工作量的任务
- 涉及多个模块的改动

---

## 对齐闸门

在需求对齐完成前，不得进入实现。

必须先明确以下三项：

- 目标：到底要解决什么问题
- 范围：这次包含什么，不包含什么
- 验收：怎样才算完成

如果以上任一项仍不清楚：

- 继续做澄清、拆解和确认
- 可以输出需求确认单、Task Session、PRD 或验收清单
- 不要直接开始编码

---

## 分析流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  1. 理解    │ → │  2. 提问    │ → │  3. 确认    │ → │  4. 输出    │
│  用户意图   │    │  澄清细节   │    │  对齐目标   │    │  需求文档   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

## 第一步：理解用户意图

### 提取关键信息

从用户的描述中提取：

```markdown
## 原始需求

用户说："我想加一个 MCP 配置页面"

## 初步理解

**核心目标**: 让用户能够管理 MCP 配置
**涉及模块**: 前端页面、后端 API、数据库
**用户角色**: 系统管理员/普通用户？
**使用场景**: 首次配置？日常管理？
```

### 识别缺失信息

| 方面 | 可能缺失 | 影响 |
|------|----------|------|
| **Who** | 谁使用这个功能？ | 权限设计 |
| **What** | 具体要做什么操作？ | 功能范围 |
| **When** | 什么时候使用？ | 性能要求 |
| **Where** | 在哪里使用？ | 兼容性 |
| **Why** | 为什么要这个功能？ | 优先级判断 |
| **How** | 希望如何使用？ | 交互设计 |

---

## 第二步：提问澄清

### 标准问题清单

**目标对齐：**
1. "这个功能主要解决什么问题？"
2. "用户完成什么任务时需要这个功能？"
3. "成功的标准是什么？"

**范围界定：**
4. "包含哪些具体操作？（增删改查？）"
5. "有什么明确不包含的吗？"
6. "是否需要兼容现有数据？"

**验收标准：**
7. "怎样算这个功能完成了？"
8. "有没有必须满足的条件？"
9. "异常情况如何处理？"

**非功能需求：**
10. "有多少用户会使用？"
11. "有没有性能要求？"
12. "安全方面有什么考虑？"

### 提问示例

**场景 1：用户说"加个 MCP 配置页面"**

```markdown
让我理解一下您的需求：

**关于用户**：
- 这个页面是给管理员用，还是普通用户也能访问？
- 同一时间大概有多少人使用？

**关于功能**：
- 需要支持哪些操作？（我猜测：查看列表、新增、编辑、删除？）
- 是否需要批量操作，比如批量导入？
- 配置项有多少个字段？有敏感信息需要加密吗？

**关于交互**：
- 编辑是弹窗形式还是跳转新页面？
- 删除需要确认吗？
- 表单验证有哪些规则？

**关于优先级**：
- 这是 blocker 吗，还是可以后续迭代？
- 期望什么时候能用上？
```

**场景 2：用户说"优化一下这个页面"**

```markdown
关于优化，我想确认几个方向：

**性能方面**：
- 现在主要慢在哪里？（加载？交互？数据处理？）
- 期望提升到什么程度？

**体验方面**：
- 用户反馈的主要痛点是什么？
- 有没有具体的交互流程觉得不合理？

**视觉方面**：
- 是样式调整，还是整体 redesign？
- 有设计稿参考吗？
```

## 第三步：确认对齐

### 需求确认模板

在正式开始前，用以下模板与用户确认：

```markdown
## 需求确认单

### 目标
{一句话描述要解决的问题}

### 范围
**包含**：
- [ ] 功能点 1
- [ ] 功能点 2

**不包含**（本次不做）：
- 功能点 3（后续迭代）
- 功能点 4（超出范围）

### 验收标准 (AC)

**AC-001**: {Given-When-Then 格式}
- Given: 前置条件
- When: 用户操作
- Then: 期望结果

**AC-002**: ...

### 非功能需求
- 性能：{要求}
- 安全：{要求}
- 兼容：{要求}

### 技术约束
- {约束 1}
- {约束 2}

### 时间节点
- 期望完成：{日期}
- 里程碑：{如果有}

---

**请确认以上理解是否正确？有需要调整的地方吗？**
```

### 对齐完成判定

只有同时满足以下条件，才算完成需求对齐，可以进入后续设计或实现：

- 目标没有歧义
- 范围边界明确
- 验收标准可验证
- 关键假设已经显式写出

若仍存在明显歧义，必须继续停留在需求分析阶段。

### Given-When-Then 格式

```markdown
## 验收准则示例

**AC-001: 新增 MCP 配置**

Given: 用户在 MCP 配置页面
When: 点击"新增配置"按钮并填写有效表单
Then: 
  - 配置保存成功
  - 页面返回列表并显示新配置
  - 显示"创建成功"提示

**AC-002: 表单验证失败**

Given: 用户在新增配置表单
When: 提交时必填字段为空
Then:
  - 对应字段显示错误提示
  - 表单不提交
  - 页面保持当前状态

**AC-003: 删除确认**

Given: 用户在配置列表
When: 点击删除按钮
Then:
  - 弹出确认对话框
  - 显示配置名称
  - 点击确认后才真正删除
```

## 第四步：输出需求文档

### 最小需求文档

即使是小功能，也建议包含：

```markdown
# 需求：{功能名称}

## 背景
{为什么需要这个功能}

## 目标
{要解决什么问题}

## 功能清单
- [ ] {功能点 1}
- [ ] {功能点 2}

## 验收标准
{AC-001...}

## 技术约束
{如果有}

## 优先级
{P0/P1/P2/P3}
```

### 完整需求文档

对于复杂功能：

```markdown
# PRD: {功能名称}

## 1. 概述

### 1.1 背景
### 1.2 目标
### 1.3 范围
- 包含
- 不包含

## 2. 用户故事

作为 {角色}，我想要 {功能}，以便 {价值}。

## 3. 功能需求

### 3.1 功能 1
**描述**: 
**输入**: 
**输出**: 
**业务规则**: 

### 3.2 功能 2
...

## 4. 非功能需求

### 4.1 性能
### 4.2 安全
### 4.3 可用性

## 5. 验收标准

{AC-001...}

## 6. 界面原型

{如果有}

## 7. 依赖与风险

### 7.1 前置依赖
### 7.2 风险
### 7.3 应对措施

## 8. 附录

### 8.1 术语表
### 8.2 参考文档
```

## 需求分级

### P0 - 关键（Must Have）
- 阻塞发布的功能
- 核心业务流程
- 法律合规要求

### P1 - 重要（Should Have）
- 显著改善体验
- 用户明确期望
- 相对独立的改进

### P2 - 期望（Nice to Have）
- 锦上添花
- 资源允许时做
- 可以延后

### P3 - 低优先级（Later）
- 未来考虑
- 不确定性高
- 影响面小

## 常见陷阱

### 陷阱 1：假设已知
```
错误： "这个需求很明显，不需要问"
正确： "即使是看起来简单的需求，也确认一下理解是否正确"
```

### 陷阱 2：范围蔓延
```
错误： "既然都做了，顺便把这个也加上吧"
正确： "这个很好，我们记为后续迭代，先保证核心功能"
```

### 陷阱 3：验收标准模糊
```
错误： "页面要好看"
正确： "页面风格与现有设计系统一致，通过设计 review"
```

### 陷阱 4：忽视非功能需求
```
错误： 只考虑功能，不考虑性能/安全
正确： "这个功能并发量大吗？有敏感数据吗？"
```

## 快速检查清单

**开始前：**
- [ ] 我理解用户要解决什么问题吗？
- [ ] 我知道这个功能给谁用吗？
- [ ] 我清楚包含哪些功能点吗？
- [ ] 我知道什么不做吗？

**分析中：**
- [ ] 我问了澄清问题吗？
- [ ] 我记录了用户的回答吗？
- [ ] 我识别了潜在风险吗？

**确认时：**
- [ ] 我总结了需求给用户确认吗？
- [ ] 验收标准可验证吗？
- [ ] 用户确认理解正确吗？

```
- AIASYS-TECH-013: 技术改进
```
