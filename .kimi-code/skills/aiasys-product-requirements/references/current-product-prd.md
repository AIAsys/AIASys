## 2. 产品定位

AIASys 是一个以任务工作区为核心的 AI 工作系统。

它不是单纯的聊天窗口，也不是以项目树、研究团队、子代理编排为用户主语义的系统。当前阶段，AIASys 的目标是让用户围绕一个可长期使用的任务工作区，持续完成资料整理、分析、研究、执行、记录与配置管理。

当前进一步收敛的目标产品语义是：

- `任务工作区` 是一等对象
- `对话` 是任务工作区内部的记忆分支
- `执行轮次` 是对话内部的运行证据单元

当前产品主线还明确要求：

- 不再以兼容旧 session-first 产品心智为目标
- 可以清理本地真实运行产生的旧对话历史后再进入新主模型
- 后续入口、命名和对象边界直接围绕任务工作区展开

一句话定位：

> AIASys 是一个以长期任务工作区为核心、允许在同一工作区下持续推进多个对话分支的 AI 工作台。

当前阶段的默认交付形态是：

> 单机个人工作模式。

也就是说，当前产品默认面向单个使用者长期使用，不把登录、租户切换和团队协作作为第一阶段主承诺。
当前运行时主线也进一步收口为：

> 先保留本地执行链路，暂停 Docker 运行时支持，把复杂度留给后续再引入。

## 3. 核心产品对象

### 3.1 任务工作区

任务工作区是当前系统的一等产品对象。

用户在一个任务工作区中长期积累：

- 对话与上下文
- 文件与资料
- 执行记录与证据
- 当前任务级配置
- 与该任务相关的知识与资源入口

当前实现层很多地方仍落在 `session / conversation` 命名上，但产品语义统一解释为任务工作区。

### 3.2 对话

对话是任务工作区内部的二级对象。

一个任务工作区下可以存在多个对话，用来承接：

- 不同阶段的推进路径
- 同一任务下的不同思路分支
- 清空记忆后重新开始的新轮次
- 仅围绕某个子问题展开的短对话

当前产品语义上：

- `新建任务` = 新建任务工作区
- `新建对话` = 在当前任务工作区内新增一个记忆分支
- `清空记忆重新开始` 默认优先解释为在当前任务工作区内新建一个空对话并切换过去
- `Fork 对话` = 基于当前或历史对话内容派生新的对话分支，并保留原对话
- 当用户还没有打开任何工作区时，`/workspace` 默认应停留在工作区首页，而不是自动跳进某个已有工作区

这三件事不能继续混成同一个动作。

需要强调的是：

- 当前代码实现层仍主要以 `session / conversation` 为运行事实源
- “工作区下多个对话”是当前主线要持续收口的产品方向
- 文档不应把它误写成所有前后端层面都已经完全完成
- 当前主线不要求为旧 session-first 产品心智保留长期兼容表达

### 3.3 用户默认配置

用户默认配置是跨任务复用的长期设置层，用于决定用户通常如何使用系统，而不是某一次任务当前怎么跑。

当前默认实现口径是：产品不强调显式登录流程，但内部仍允许保留单个默认用户骨架，以兼容现有数据目录、知识库与工作区隔离实现。

### 3.4 市场目录

市场目录表达的是系统供给侧目录，即系统支持哪些 Skill、MCP 或其他能力接入项。

市场不等于当前正在使用的能力清单，也不等于用户默认配置。

### 3.5 运行证据

运行证据用于回答“这一轮实际使用了什么”，包括当前执行所采用的配置、能力、运行环境与执行记录。

当前更清楚的对象层级应为：

- 工作区层：长期任务资产、任务级配置与新分支默认执行绑定
- 对话层：记忆与推进路径，以及当前分支的临时执行覆盖
- 执行轮次层：工具调用、运行记录和配置快照

### 3.6 三层作用域模型

AIASys 当前不引入“全局工作区”作为主对象。

更稳定的模型是：

- 系统目录层
  - 回答“系统提供什么”
- 用户默认层
  - 回答“用户长期默认怎么用”
- 任务工作层
  - 回答“这个任务当前实际启用了什么”

因此：

- 全局层可以有市场目录、数据库连接器本体、知识库本体、知识图谱本体
- 用户默认层可以有默认 Agent 配置、长期偏好、默认 MCP 候选
- 任务工作层负责挂载、启用、覆盖和当前任务级资源选择

### 3.6A 当前分支覆盖层

在任务工作层内部，当前主线还需要继续区分：

- 工作区层
  - 回答“这个长期任务工作区整体怎么组织”
- 当前分支层
  - 回答“当前这条分支这次具体怎么跑”

因此后续对外入口必须明确：

- `工作区设置`
  - 负责工作区标题、描述、新分支默认模式、共享资源挂载与默认执行方式
- `当前分支设置`
  - 负责当前分支 mode、当前分支 Agent 覆盖与当前分支运行态应用
- `我的默认设置`
  - 负责用户长期默认 Agent 配置

当前主线还需要补一条更明确的口径：

- 工作区层的“默认执行方式”
  - 回答“这个工作区后续新分支默认怎么跑”
- 当前分支层的“运行态覆盖”
  - 回答“这条分支这次临时改成怎么跑”
- 执行轮次层的“运行证据”
  - 回答“这一轮实际上用了什么执行方式和运行环境”

当前代码里历史上仍保留 `sandbox_mode / env_id` 等实现字段，但用户可见语义不应再把它们直接包装成一个脱离任务上下文的独立市场主概念。

## 4. 用户价值

当前阶段，AIASys 主要提供以下用户价值：

1. 用一个长期任务工作区承接复杂工作，而不是把每次工作拆成相互断裂的会话。
2. 把当前任务配置、用户默认配置和系统供给目录清楚分层，降低配置混乱成本。
3. 支持在同一个任务工作区内按需要新建多个对话，而不是把所有历史都硬塞在一条长对话里。
4. 支持通过多种能力接入方式扩展工作能力，而不是把所有外部能力都绑死在单一协议上。
5. 保留执行记录和工作证据，便于用户追溯任务实际发生了什么。

## 5. 当前产品原则

### 5.1 任务优先

当前产品主语义是任务工作区，而不是项目空间、研究团队或功能广场。

### 5.1A 工作区比对话更长寿

任务工作区是长期对象，对话是工作区内部的可增删、可分支、可重开的记忆单元。

因此：

- 清空记忆不等于删除工作区
- 新开对话不等于新建任务
- 新建任务才等于新建工作区

### 5.1B 新建任务与新建对话的命名必须分层

当前主线不建议把 `新建工作区` 和 `新建任务` 做成两个并列主按钮。

更稳定的命名口径是：

- 一级主按钮叫 `新建任务`
  - 其真实含义是创建新的任务工作区
- 当前工作区内的二级按钮叫 `新建对话`
  - 其真实含义是在当前工作区中新增一个对话分支
- 如果需要显式暴露快捷动作，可补 `清空记忆并新建对话`
- 如果需要保留分支能力，可补 `Fork 对话`

这样用户操作词是 `任务 / 对话`，系统结构词是 `工作区`，两层语义不会打架。

### 5.2 分层优先于堆功能

AIASys 的核心问题优先级是：

- 先把信息层级讲清楚
- 再决定操作入口怎么放
- 不把多个作用域混在同一个页面中解释

### 5.3 当前真相与目标态分开写

如果某个能力尚未完全实现，文档必须明确写清：

- 当前已经提供什么
- 后续准备扩展什么
- 中间还缺哪些能力或约束

### 5.3A 当前主线不做兼容迁移包装

本轮产品重构已经明确：

- 不继续为旧的 `session-first` 产品语义设计长期兼容层
- 可以清理本地真实运行历史，再进入新的工作区/对话主模型
- 新增入口、命名、接口与说明文案都应优先服务新模型

这意味着：

- 旧历史如果仅构成重构负担，可以被清理
- 新建任务、新建对话、Fork 对话不应再退化回旧“新建会话”心智
- 文档不能再把“兼容旧会话模型”写成默认路线

### 5.4 系统提供不等于强制启用

系统级目录项表示“系统提供”，不表示“所有用户默认启用”。

## 6. 信息架构

### 6.1 任务工作区层

任务工作区承接当前任务的实际工作状态，主要包含：

- 对话区
- 当前工作区下的对话列表
- 右侧任务工作区
- 当前任务配置
- 文件、知识、产物等任务内资源

当前右侧默认应只承接“任务工作区”本身，而不是把执行流、数据库、知识图谱等对象并列成多个一级常驻主区。

当前更稳的结构是：

- 左侧先展示任务工作区，再按工作区展开或切换其下对话
- 中间主区始终只显示当前选中工作区中的当前对话
- 右侧默认展示文件、资源、产物和工作区摘要
- 执行记录通过抽屉或二级入口进入
- 当前分支设置通过当前对话上下文中的弹窗进入
- 工作区设置通过工作区上下文中的弹窗进入

### 6.1A 当前分支设置

当前分支设置不是工作区设置的一个 tab，而是单独的工作入口。

它负责：

- 当前分支 mode
- 当前分支 Agent 覆盖
- 当前分支工具切换
- 当前分支运行态应用与重置

它不负责：

- 工作区标题与描述
- 工作区共享知识与资源挂载
- 用户长期默认配置

### 6.2 设置层

设置层承接用户长期默认配置和系统级目录入口，不直接承载当前任务的一次性工作状态。

当前设置层的目标语义包括：

- 用户默认 Agent 配置
- 模型与偏好
- 运行环境管理入口
- Skill 市场
- MCP 市场
- 后续变量与密钥管理入口

这里的 `Agent 配置` 明确只承接：

- 用户长期默认配置

不再直接冒充：

- 当前分支设置
- 工作区设置

当前设置层在产品上不再依赖“登录后进入个人中心”的表达，而是直接作为单机个人工作台里的长期默认配置入口存在。

### 6.3 市场层

市场层是供给侧目录，负责展示系统支持什么。

当前已经明确存在两类市场：

- Skill 市场
- MCP 市场

后续如果出现更多能力目录，应继续按目录层方式扩展，而不是直接塞回任务配置弹层。

### 6.4 资源与能力入口

资源与能力入口用于承接工作区相关资源的访问路径，例如知识库、知识图谱、环境管理等，但不应重新演化成第二套系统主导航。

## 6.5 任务工作区生命周期与任务配置边界

当前主线补充收口如下：

- 单个任务工作区主生命周期只表达 `draft / active / completed / deleting / deleted`
- 任务工作区内部允许存在多个对话，对话生命周期不等同于工作区生命周期
- 执行状态与运行态状态必须独立于任务主状态
- 新建任务只负责启动必需项，不负责完整配置
- 任务配置默认挂在工作区层，负责当前任务级 Agent、能力、环境、资源覆盖
- 新建对话与清空记忆默认不应重置工作区层配置
- 当前运行中保存任务配置，默认对下一次执行生效

---

## 6.6 单机个人模式与默认用户边界

当前主线补充收口如下：

- 默认交付形态是单机个人工作模式
- 默认本地用户只承担兼容骨架职责
- 当前文档与前端优先突出任务工作区，而不是用户体系本身
- 多人登录、团队共享与租户语义不作为当前主线默认承诺

## 7. 配置分层

AIASys 当前明确区分三层配置。

### 7.1 系统目录层

回答：系统支持什么。

典型对象：

- Skill 市场条目
- MCP 市场条目
- 系统集成目录项

### 7.2 用户默认层

回答：用户长期默认使用什么。

典型对象：

- 用户默认 Agent 配置
- 用户默认模型与偏好
- 我的 MCP
- 后续变量与密钥

当前补充口径：

- 用户长期偏好记忆也属于这一层
- 这层回答的是“通常默认怎么用”
- 不直接等于“当前任务已经启用什么”

### 7.3 任务工作层

回答：当前任务这次实际启用什么。

典型对象：

- 任务级 Agent 配置
- 任务级能力覆盖
- 任务级运行环境选择
- 当前工作区下的默认对话推进方式
- 任务级资源挂载

当前推荐把下列对象解释为“资源本体 + 工作区挂载关系”：

- 数据库
- 知识库
- 知识图谱

也就是说：

- 数据库连接器本体属于系统目录层或长期资源层
- 知识库本体属于系统目录层或长期资源层
- 知识图谱当前真相仍是系统级单图谱，目标态应演进到多图谱本体
- 当前任务工作层决定挂载哪些数据库、知识库和知识图谱
- 默认不自动挂载知识库、知识图谱和数据库连接；只有用户在当前任务工作区显式挂载后，资源才进入任务上下文
- 左侧资源入口默认应进入资源本体目录；工作区挂载视图需要显式带 `workspace_id`，不能靠隐式 URL 继承偷偷切换语义

当前产品要求：任务级配置可以动态保存，但不承诺对正在运行的 Agent Session 做热更新。默认语义是保存后对下一次执行生效。

当前主线下，对话默认不单独继承一套完整配置层。

也就是说：

- 工作区层负责“这个任务通常怎么跑”
- 对话层负责“这次从哪段记忆和路径继续推进”

如果未来需要对话级实验配置，再单独扩展，不作为当前第一阶段主线。

## 8. 能力接入模型

AIASys 不把 MCP 视为总抽象，而把“能力接入”视为一等概念。

当前至少承认三类能力接入方式：

### 8.1 CLI / 本地命令

适用于已有成熟命令行工具链的能力。

### 8.2 原生 API / SDK

适用于官方已经提供稳定 API 或 SDK 的能力。

### 8.3 MCP

适用于需要标准工具协议封装、复用现有 MCP 生态，或在多 Agent 之间共享工具暴露层的场景。

因此当前产品原则是：

- MCP 是一种接入方式，不是所有能力的总名称
- MCP 市场描述的是协议型接入目录
- 不是所有外部能力都必须先进 MCP 市场

当前补充口径：

- MCP 条目目录和长期候选配置可以存在于系统目录层与用户默认层
- 但当前任务是否启用哪些 MCP，属于任务工作层
- 任务级 MCP 配置保存后默认下一次执行生效

### 8.4 Skill 的当前产品定义

Skill 在当前产品里不应被理解为“单个可编辑的 `SKILL.md` 文本”。

更准确的定义应是：

- 一个可安装的 skill 目录包
- 其中 `SKILL.md` 只是入口文件
- 目录内部可以是任意目录树
- `references`、`templates`、`scripts`、`assets` 只是常见示例，不是协议要求
- 后续允许通过压缩包等形式导入安装

因此后续 Skill 市场应优先支持：

- 查看系统提供的 Skill 目录项
- 将内置 Skill 或 zip 包导入到当前工作区
- 查看当前工作区已经拥有的 Skill 包
- 删除当前工作区中的 Skill 包
- 需要时查看入口 `SKILL.md`

当前产品不再围绕“在线编辑单个 MD 文件”设计，也不再把 Skill 作为用户级长期共享对象来建模。

当前第一阶段建议把 Skill 实际存放在任务工作区根目录下的可见目录，例如：

```text
workspaces/{user_id}/{workspace_id}/skills/{skill_package}/
```

这样用户可以直接在工作区看到自己导入的 Skill 包，系统运行时也直接挂载这一真实目录，而不是再维护一层隐藏的用户级共享副本。

同时应区分两层：

- Skill 市场与系统提供的 Skill 目录项属于系统目录层
- 当前任务工作区是否拥有和启用哪些 Skill 包，属于任务工作层

当前实现还应明确一条落地约束：

- 运行时系统目录应来自独立的 builtin Skill 目录，例如 `apps/backend/builtin_skills/`
- `.agents/skills/` 属于 AIASys 的协作/开发 skill 仓库，不应直接作为用户运行时 Skill 市场来源

Skill 与 MCP 的当前生效语义也需要统一：

- 允许用户在当前任务工作区中修改 Skill 包和 MCP 配置
- 修改会先保存到工作区
- 当前已经启动的运行态不会在 turn 进行中热更新
- 系统会把当前任务标记为“待刷新运行态”
- 下一次执行前，系统重建当前任务的运行态实例，再加载最新的 Skill 和 MCP 配置

这里的“重建运行态”不等于新建任务，也不等于清空历史。它只是替换当前任务背后的活跃 Agent runtime，任务本身的对话历史、文件和工作区内容仍然保留。

---

## 8.5 当前产品模式

当前产品模式收口为：

- 默认单机个人使用
- 不把登录/登出作为主产品流程
- 不把团队级、个人级、任务级多层继承作为当前第一阶段复杂度来源

内部如保留 `user_id`，应视为实现兼容骨架，而不是当前产品主语义。

补充一条目标态收口：

- 不再把 `数据分析模式 / 深度研究模式` 作为长期一级产品模式
- 顶层产品默认只有一个助手入口
- 分析、研究等差异优先以下列对象表达：
  - 内置工具包
  - 任务预置
  - 默认提示词与策略 overlay
  - 专业子角色

---

## 9. 深度研究模式的当前定义

当前代码事实里，深度研究模式的定位已经收敛为：

> 面向论文阅读与研究整理的任务模式。

它当前主承诺的能力是：

- 搜索
- 阅读
- 翻译
- 记录
- 整理

当前不把“方法复现”作为默认承诺。

同时，`research mode` 当前只表示默认推荐和能力预置，不表示硬性能力上限。

目标态进一步收口为：

- `research mode` 不再作为长期一级产品模式保留
- 它未来应下沉为统一助手下的 `research-pack` 或同类内置工具包
- 对用户暴露的主语义应是“当前任务使用了哪些能力增强”，而不是“我切换到了另一个模式”

## 10. 设置与市场的当前要求

### 10.1 设置不是单页，而是一层长期默认配置系统

设置不应继续和任务即时配置混在一起。

### 10.2 市场不等于当前启用状态

市场页展示的是目录和供给情况，不是当前任务已经启用的列表。

### 10.3 任务配置不承担市场职责

任务配置应回答“这个任务现在怎么跑”，不承担系统目录展示和长期默认配置中心职责。

## 11. 关键功能需求

### FR-01 任务工作区主语义

系统必须以任务工作区作为当前一等产品对象，并围绕该对象承接对话、文件、执行记录和任务级配置。

### FR-01A 工作区下多对话

系统应支持单个任务工作区下存在多个对话。

这些对话共享同一个任务工作区的文件、资料、产物和任务级配置，但各自维护独立的记忆推进路径与对话历史。

### FR-01B 清空记忆与新建任务分离

系统必须把以下动作严格区分：

- 新建任务：创建新的任务工作区
- 新建对话：在当前任务工作区内创建新的对话
- 清空记忆重新开始：默认优先实现为在当前任务工作区内创建一个新的空对话，而不是删除工作区资产
- Fork 对话：从当前或历史对话内容派生新的对话分支，原对话保留不变

### FR-01C 对话分支能力

系统应支持基于已有对话内容创建分支对话。

当前可接受的实现方式是：

- 复制当前对话或指定历史截点之前的上下文
- 在同一工作区内创建新的对话
- 保留原对话不被覆盖

这类能力应被解释为“工作区内的新对话分支”，而不是“新建任务”。

### FR-02 配置分层清晰

系统必须区分系统目录层、用户默认层和任务工作层，避免多个作用域混写。

### FR-03 统一任务配置入口

系统应提供统一的任务配置入口，至少承接：

- Agent 配置
- 能力配置
- 运行环境配置

### FR-04 设置中心承接长期默认配置

系统应把用户级默认配置收口到统一设置层，而不是继续散落在任务工作台主链中。

### FR-05 市场页承接供给目录

系统应通过市场页展示系统当前支持的 Skill 和 MCP 条目，并允许用户进一步安装或配置。

### FR-05A 主界面壳层稳定

系统主界面必须优先稳定为：

- 左侧任务导航
- 中间主工作区
- 右侧任务工作区

其中执行记录与诊断信息不应与右侧任务工作区并列做成一级常驻主区。

### FR-06 能力接入模型可扩展

系统必须允许能力通过 CLI、原生 API / SDK、MCP 等多种方式接入，而不是强制绑定单一协议。

### FR-07 任务级配置 next-run-only

对于任务级 Agent 配置和任务级 MCP 调整，系统必须明确支持保存，但默认对下一次执行生效，而不是运行中热更新。

### FR-08 运行证据可追溯

系统应能回答这一轮执行实际使用了什么配置、能力和运行环境。

## 13. 当前验收口径

一份符合当前主线的实现，至少应满足：

- 用户能理解任务工作区是当前主对象
- 用户能理解同一个任务工作区下可以有多个对话
- 用户能区分“新建任务”和“新建对话/清空记忆重新开始”
- 用户能区分设置、市场、任务配置三类入口
- 用户能理解哪些是系统支持，哪些是我的默认，哪些是当前任务实际使用
- 用户能看到或追溯当前任务的执行记录与关键运行证据
- 文档、前端入口和后端作用域模型不互相打架
