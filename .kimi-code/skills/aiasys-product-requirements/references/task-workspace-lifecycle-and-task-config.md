# 任务工作区生命周期与任务配置产品设计

状态: 主线专题
更新时间: 2026-04-05
适用版本: v0.3.9 当前主线

---

## 1. 文档目标

本文档用于收敛三个容易混淆的产品问题：

1. 单个任务工作区的生命周期是什么
2. 新建任务时到底需要用户先决定什么
3. 进入任务后，任务配置负责什么

本文档属于产品语义文档，不替代具体接口字段和状态机实现。

当前主线补充约束：

- 不继续为旧 session-first 语义设计长期兼容迁移
- 可以在进入新模型前清理本地真实运行历史
- 后续接口与交互动作优先围绕 `任务工作区 / 对话 / 执行轮次` 展开

---

## 2. 任务工作区与对话的生命周期关系

### 2.1 对象层级

当前主线应把三层对象区分开：

- 任务工作区
- 工作区下的对话
- 对话中的执行轮次

其中：

- 任务工作区是长期对象
- 对话是工作区内部的记忆分支
- 执行轮次是某次具体运行的证据单元

### 2.2 主生命周期

单个任务工作区当前应只表达以下主状态：

- `draft`
- `active`
- `completed`
- `deleting`
- `deleted`

这些状态表达的是任务工作区本身的生命周期，而不是某一轮执行结果。

### 2.3 对话生命周期默认独立于工作区生命周期

对话不应复用工作区的主生命周期。

对话更适合作为工作区内部的可切换、可新增、可归档对象，例如：

- 当前对话
- 历史对话
- 新建空对话
- 已归档对话

也就是说：

- 结束一个对话不等于结束整个任务工作区
- 清空记忆不等于删除整个任务工作区
- 新建任务也不等于只是在当前工作区里开个新对话
- Fork 对话也不等于新建任务，它只是当前工作区中的分支对话

### 2.4 执行状态与任务主状态分离

任务工作区的主状态必须与执行状态分离。

执行状态用于表达当前一轮是否正在运行，例如：

- `idle`
- `running`
- `waiting_user`
- `succeeded`
- `failed`
- `cancelled`

执行失败不应直接把任务工作区打成“永久失败”。

### 2.5 运行态状态与任务主状态分离

任务工作区的主状态也必须与运行态状态分离。

运行态状态用于表达当前 runtime 是否可立即执行，例如：

- `not_initialized`
- `ready`
- `busy`
- `refresh_required`
- `missing`

这层状态回答的是“当前能不能直接跑”，不是“这个任务是否结束”。

### 2.6 completed 的当前语义

`completed` 当前表达的是：

- 上一轮执行已经结束
- 这个任务工作区可以继续被使用
- 下一次启动执行时，会重新进入 `active`
- 工作区下仍然允许继续新增或切换对话

它不表达“任务永久关闭”。

## 3. 新建任务与任务配置的边界

### 3.1 新建任务只决定启动姿势

新建任务的目标是让用户快速进入一个可用任务工作区。

因此新建任务默认只决定启动必需项：

- 任务模式
- 初始运行环境
- 初始化方式

当前主线对“初始运行环境”的推荐解释进一步收口为：

- 默认不再让用户先选 Docker 容器
- 默认使用工作区本地 Python 执行链路
- 工作区 Python 环境物料优先由 `uv` 管理
- 只有在用户明确需要时，才暴露更细的 Python/依赖初始化控制

当前新建任务的语义必须进一步明确为：

- 创建新的任务工作区
- 自动创建该工作区的首个默认对话
- 进入这个默认对话开始工作

当前推荐的动作命名是：

- `新建任务`
- `新建对话`
- `清空记忆并新建对话`
- `Fork 对话`

不建议把 `新建工作区` 与 `新建任务` 做成两个并列主按钮。

### 3.2 新建任务不应变成大表单

以下内容不应作为新建任务的默认必填项：

- 完整任务级 Agent 配置
- 完整 MCP 列表
- Skill 包安装与卸载
- 详细资源挂载
- 大量高级策略开关

原因很简单：

- 创建流程会过重
- 生命周期边界会模糊
- 用户往往进入任务后才知道要不要改

### 3.3 工作区设置与当前分支设置必须分开

任务创建成功后，后续配置入口不能继续只叫一个笼统的“工作区配置”。

更稳定的模型是：

- `工作区设置`
  - 负责长期工作区信息与共享资源
- `当前分支设置`
  - 负责当前分支这次实际如何执行

两者需要回答的问题不同：

- 工作区设置回答：
  - 这个工作区叫什么
  - 这个工作区的新分支默认模式是什么
  - 这个工作区共享挂载了哪些资源
- 当前分支设置回答：
  - 当前分支用什么 mode
  - 当前分支覆盖了哪些 Agent 提示词和工具
  - 当前分支当前运行态是否需要刷新
  - 当前分支这次修改是立即生效还是下次执行生效

当前主线推荐：

- 工作区级配置继续负责共享资源与默认值
- 当前分支级配置负责执行预置和 Agent 覆盖
- 用户默认层负责跨工作区复用的长期默认 Agent 配置

## 4. 工作区设置与当前分支设置的产品范围

进入任务工作区后，至少需要区分两类配置对象：

- 工作区设置
- 当前分支设置

### 4.1 工作区设置

- 工作区标题
- 工作区描述
- 新分支默认模式
- 工作区共享资源挂载
- 工作区共享能力摘要

### 4.2 当前分支设置

- 当前分支 mode
- 当前分支 Agent 覆盖
- 当前分支提示词覆盖
- 当前分支工具启用摘要
- 当前分支运行态应用与重置

### 4.3 对话层默认负责什么

对话层当前默认负责：

- 当前记忆上下文
- 当前推进路径
- 当前对话历史
- 与该对话关联的执行证据入口
- 从现有对话内容 Fork 出新的分支对话

对话层当前默认不负责：

- 独立复制一套完整任务配置
- 单独复制一份工作区文件树
- 重定义任务工作区本身的生命周期

但对话层需要允许：

- 保存当前分支级 Agent 覆盖
- 保留“当前分支 mode 是什么”的执行事实
- 在当前分支已经有聊天或执行记录时锁定 mode，改为新建另一条分支

## 5. 生命周期阶段下的产品约束

### 5.1 draft

`draft` 阶段允许最大范围编辑：

- 基础信息可编辑
- Agent 可编辑
- 能力可编辑
- 环境可编辑
- 资源可编辑

### 5.2 active

`active` 阶段仍允许保存任务级配置，但必须区分“保存”和“当前运行热更新”：

- 任务级 Agent 配置允许保存，但默认下一次执行生效
- 任务级能力配置允许保存，但默认下一次执行生效
- 运行环境允许调整，但可能要求等待或重建 runtime

同一工作区下如果新建对话：

- 默认复用当前工作区已有任务级配置
- 不因新对话而清空工作区文件、产物和资源挂载
- 当前对话的记忆与历史从空开始

同一工作区下如果 Fork 对话：

- 默认复用当前工作区已有任务级配置
- 保留原对话不变
- 新对话继承被选中范围内的历史上下文
- 新对话后续的执行记录独立沉淀

### 5.3 completed

`completed` 不等于关闭。

因此 `completed` 阶段仍允许调整任务配置，下一次执行重新进入 `active`。

---

## 6. 当前产品结论

当前主线必须坚持以下结论：

1. 单个任务工作区必须明确分层：任务主状态、执行状态、运行态状态。
2. 单个任务工作区下允许存在多个对话，对话生命周期独立于工作区主生命周期。
3. 新建任务 = 新建工作区并创建首个默认对话，不等于仅新开一条对话。
4. 清空记忆重新开始默认优先实现为在当前工作区中创建新的空对话，而不是删除工作区资产。
5. Fork 对话属于当前工作区中的分支能力，不等于新建任务。
6. 任务配置是当前工作区级覆盖中心，不是全局默认配置中心，也不是每个对话都单独复制一套。
7. 当前运行中允许保存配置，但默认不承诺热更新。
