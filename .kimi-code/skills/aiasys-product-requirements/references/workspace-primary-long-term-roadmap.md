# 工作区主线长程执行台账

状态: 主线台账
更新时间: 2026-04-06
适用版本: v0.3.9 当前主线

---

## 1. 文档目标

本文档用于把 AIASys 当前“任务工作区优先”主线整理成一份可持续推进的长期 todo，避免每次实现都从零对齐方向。

它主要回答 4 个问题：

1. 当前主线已经完成了哪些关键收口
2. 后续还缺哪些批次任务
3. 这些任务应该按什么顺序推进
4. 在什么情况下需要暂停并重新对齐，而不是继续自动实现

本文档不是单次 task session 的替代品。

更合适的关系是：

- 本文档负责长期路线、批次顺序和验收口径
- 当前 active task session 负责某一批次的局部实现和证据沉淀

## 3. 长程执行规则

为了减少反复打断，后续推荐按“批次”而不是按“零碎点”推进。

### 3.1 推荐执行方式

每一轮只选一个批次推进，直到同时满足：

- 该批次的核心验收项完成
- 代码与文档口径一致
- 最低限度的 build / route test / browser smoke 完成

不要在同一轮中同时横跳多个未收口的批次。

### 3.2 允许自动继续的前提

只要没有碰到以下 blocker，就应沿当前批次持续推进，不必频繁停下来重新问方向：

- 需要用户提供新的产品边界
- 会影响大量现有示例数据或真实历史数据
- 会引入明显的模型反转，例如从工作区优先退回会话优先
- 需要外部服务凭据、基础设施权限或真实生产信息

### 3.3 必须暂停的情况

出现以下情况时应暂停，并回到用户确认：

- 当前批次内部出现两个互斥产品方向
- 某个改动需要删除一大块尚不确定是否遗留的旧能力
- 某个入口的“当前真相”与既有 PRD 明显冲突
- browser smoke 显示用户路径已经回归，但短时间无法定位根因

## 4. 主台账

## Batch A: 语义与信息架构收口

状态: 已完成（2026-04-06）
优先级: P0

### 目标

把目前界面中还残留的旧 `session-first / execution-first` 口径收掉，让工作区作为稳定主对象真正贯穿左中右三栏和主要弹窗。

### 包含

- 左侧历史树、空态和菜单中的旧会话优先文案收口
- 中间欢迎页、状态提示、生命周期弹窗中的旧口径收口
- 删除或降级仍然把“执行流”当一级主入口的表达
- 统一 `新建任务 / 新建对话 / Fork 对话 / 清空记忆并新建对话` 的产品说明

### 不包含

- 不重做左侧视觉风格
- 不重构执行流内部事件模型

### 验收标准

- 用户可见主入口不再混用 `工作区 / 会话 / Session / 执行空间` 多套主语义
- 左侧、欢迎页、顶栏、生命周期弹窗口径一致
- `/workspace` browser smoke 能完整走通工作区选择、分支选择、工作区配置打开

### 建议拆分

1. 左侧历史树和空态
2. 中部欢迎页和状态提示
3. 生命周期与记录弹窗口径

## Batch B: 主画布多对象预览

状态: 进行中（已完成文件主画布、PDF/聊天产物打开、主图谱过渡入口）
优先级: P0

### 目标

把中间主画布从“仅能打开部分文件”扩展为工作区对象的大画布承载区，让更多工作区资产不必挤在右栏里预览。

### 包含

- PDF 预览
- 图片预览
- 图表与报告类对象预览
- 图谱对象进入主画布的过渡入口
- 从右栏、消息附件、工具产物进入主画布的统一打开动作

### 不包含

- 不把聊天永久移到右侧
- 不额外造第二套预览渲染架构

### 验收标准

- 至少 4 类对象可以从工作区上下文进入主画布
- 所有对象都能从主画布返回当前分支对话
- 复用现有预览链路，不新增重复 renderer 体系

### 当前进展

- 已完成：基础文件主画布预览
- 已完成：PDF 预览与聊天工作区产物统一“在主画布打开”
- 已完成：主图谱进入主画布的过渡入口
- 已完成：执行记录 `result_preview` 报告进入主画布
- 已修复：GraphPage 在重复加载时误报超时，避免主画布图谱出现假错误
- 下一步：把研究工作节点汇报、报告类对象继续收成统一工作区对象，而不是继续堆单点入口

### 依赖

- 需要沿用现有 `FilePreviewPanel` / `PreviewFile` 管线

## Batch C: 工作区配置页分层重构

状态: 下一优先级
优先级: P0

### 目标

把当前“大而全”的工作区配置弹窗收成更稳定的任务级配置中心，明确基础信息、Agent、能力、资源、运行态之间的边界。

### 包含

- 工作区基础信息层
- Agent 配置层
- 能力接入层
- 资源挂载与验活层
- 运行态状态与 next-run-only 提示
- 把重信息块做默认折叠，避免全部铺开

### 不包含

- 不把市场目录直接并入任务配置
- 不把长期默认设置并入任务配置

### 验收标准

- 用户能看清“当前工作区启用了什么”和“系统提供什么”不是一回事
- 资源、能力、运行态三层不再互相解释彼此
- 保存后生效时机在界面里表达清楚

### 当前补充说明

当前配置页已经能改 mode、资源挂载和部分运行态动作，后续重点是结构和层次，而不是再继续堆字段。

## Batch D: 知识与图谱跨页面工作区绑定闭环

状态: 中期主线
优先级: P1

### 目标

把知识库、知识图谱与当前工作区上下文的关系彻底做成闭环，不再出现“分析页是工作区语义，跳到知识页又丢上下文”的割裂体验。

### 包含

- 从分析工作台进入知识图谱时保留 `workspace_id`
- 图谱页显示当前工作区上下文与主图谱状态
- 知识库入口与工作区挂载摘要联动
- 多知识库 / 多图谱挂载后的最小选择与主图谱切换
- 默认资源目录页仍可管理资源本体，不因为当前分析页已有 `workspace_id` 就自动切成“当前工作区挂载视图”
- 工作区首页在未打开任何工作区时保留“最近工作区 / 新建任务”默认态，而不是强制自动打开第一个工作区

### 不包含

- 不做新的独立 `ResearchProject` 大骨架
- 不把知识页重新设计成独立门户

### 验收标准

- 用户从工作区进入知识页面时不会丢失当前任务上下文
- 图谱页能明确回答“当前命中的主图谱是谁”
- 至少一条 KB / KG -> 工作区 -> 分支的连续路径可真实 smoke

## Batch E: 分支生命周期与执行证据收口

状态: 中期主线
优先级: P1

### 目标

把“分支是工作区内的记忆分支，执行证据是分支内的运行轮次”这一层关系继续做实，避免生命周期动作和记录入口继续混淆。

### 包含

- `新建对话 / Fork / 清空记忆并新建对话 / 删除分支` 的边界说明与实现校对
- 当前分支、历史分支、已归档分支的展示规则
- 执行记录的二级入口与只读证据区定位
- 分支重命名、切换、删除后的回退路径统一

### 不包含

- 不重做 execution journal 存储模型
- 不把执行记录重新升回一级主入口

### 验收标准

- 生命周期动作不会误伤工作区本体
- 用户能区分“切分支”和“开新任务”
- 记录弹窗与右栏执行记录口径一致

---

## Batch F: 设置中心与市场目录分层

状态: 中期主线
优先级: P1

### 目标

把设置、市场、任务配置三类入口的长期分层做稳，减少系统入口继续膨胀成“万能侧栏”。

### 包含

- 设置中心只承接长期默认配置
- 市场目录只承接供给侧目录
- 任务配置只承接当前工作区实际启用内容
- 左侧系统入口精简与分组
- Skill / MCP / 变量与密钥等入口边界明确

### 不包含

- 不在这一批次引入多人协作或团队级配置层

### 验收标准

- 用户能区分“系统支持什么”和“当前任务正在使用什么”
- 任务工作台不再承担系统设置中心职责
- 市场目录不再被误解成任务当前配置页

## Batch G: 质量门禁与验收包

状态: 持续建设
优先级: P0

### 目标

建立一套稳定、可重复的验收包，避免每轮都临时拼测试和截图。

### 包含

- 关键 route 测试
- 关键 build 基线
- 浏览器 smoke 脚本
- 示例工作区 seed 与验收材料
- `/workspace` 主路径截图与行为证据

### 不包含

- 不要求一开始就做到全量 e2e
- 不做大而全的 CI 编排设计

### 验收标准

- 每个批次至少有一套可复跑的验收命令
- 关键示例工作区数据可恢复到稳定初始状态
- 后续实现不再高度依赖人工口述回忆

## 5. 推荐执行顺序

当前更合适的顺序是：

1. Batch A：语义与信息架构收口
2. Batch B：主画布多对象预览
3. Batch C：工作区配置页分层重构
4. Batch D：知识与图谱跨页面工作区绑定闭环
5. Batch E：分支生命周期与执行证据收口
6. Batch F：设置中心与市场目录分层
7. Batch G：质量门禁与验收包，贯穿全程同步补齐

原因很简单：

- A/B/C 决定主界面骨架是否稳定
- D/E 决定工作区模型是否真正可持续使用
- F 决定系统入口是否能长期扩展而不重新混乱
- G 决定后续能否少打断、少返工

## 7. 暂停条件

后续自动推进时，只有在以下条件出现时才建议暂停问用户：

1. 需要新增或删除一等产品对象
2. 需要大规模清理现有示例数据或历史运行目录
3. 需要引入新的外部系统前提，例如新的部署、鉴权或权限模型
4. 当前批次验收失败，且短时间内无法通过代码和真实 smoke 自行定位

除此之外，应默认沿当前批次继续推进，不必频繁打断用户。

## 8. 本文档的维护规则

后续只要出现以下任一情况，就应更新本文档：

- 某个批次已经完成
- 某个批次顺序需要调整
- 某个批次新增了明确 blocker
- 某个批次的范围或验收口径发生变化

推荐同步更新位置：

1. 当前 active task session
2. 本文档
3. `current-product-prd.md` 中的主稿引用入口
