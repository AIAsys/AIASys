# 新建任务与任务配置字段矩阵

状态: 主线专题
更新时间: 2026-04-04
适用版本: v0.3.9 当前主线

---

## 1. 文档目标

本文档用于把“新建任务”和“任务配置”两类入口的字段边界彻底写清楚。

它回答的是：

- 哪些字段属于新建任务阶段
- 哪些字段属于工作区设置
- 哪些字段属于当前分支设置
- 哪些字段在 `draft / active / completed` 下允许编辑
- 哪些字段修改后下一次执行生效

---

## 2. 核心结论

### 2.1 新建任务只保留启动必需项

新建任务入口只应保留：

- 模式
- 初始环境
- 初始化方式

### 2.2 工作区设置与当前分支设置分开承接

- 工作区设置承接长期工作区信息和共享资源
- 当前分支设置承接当前分支 mode、Agent 覆盖和运行态应用
- 用户默认设置承接长期个人默认 Agent 配置

### 2.3 当前运行中允许保存，不默认热更新

`active` 状态下，大部分关键字段允许保存，但默认只对下一次执行生效。

## 4. 设计解释

### 4.1 为什么“新分支默认模式”和“当前分支 mode”要分开

模式当前至少承接两类职责：

- 工作区层的“后续新分支默认进入什么预置”
- 当前分支层的“这条分支本次按什么预置执行”

因此：

- 创建工作区时要能选默认模式
- 工作区设置中允许改后续新分支默认模式
- 当前分支设置中允许在空白分支时改当前分支 mode
- 一旦当前分支已有聊天或执行记录，就不再原地切 mode，而是新建另一条分支

### 4.2 为什么“初始环境”和“初始化方式”不继续放在任务配置

这两个字段更偏“任务是怎么开始的”。

任务创建完成后，真正需要持续调整的是“当前环境”，而不是“初始环境”和“初始化方式”本身。

### 4.3 为什么资源挂载继续放在工作区设置

资源挂载本质上是工作区共享事实源，更适合长期绑定在工作区上，而不是被每条分支都复制一份。

### 4.4 为什么资源挂载在 `active` 时也允许改

因为很多任务会在执行过程中逐步补材料。

但是否立即对当前正在运行的一轮执行可见，应由 runtime 刷新策略决定，而不是在产品文档里假定为热更新。

---

## 5. 当前建议结论

当前主线应统一以下规则：

1. 新建任务只负责轻量启动。
2. 工作区设置与当前分支设置必须拆开。
3. `draft / active / completed` 三个阶段都允许继续配置，但 `active` 默认不承诺热更新。
4. 前端必须明确告诉用户哪些修改是立即生效，哪些是下次执行生效。
